# 字段建模反模式

## 何时读取

遇到这些情况时读这份 reference：

- 想知道为什么系统“能算，但不好维护”
- 看板、筛选、提醒、工作流总是要绕路
- 用户经常说“这个状态看不出来”
- 需要决定该不该新增一个显式状态字段

## 先记住一个原则

指标字段和状态字段不要混成一类。

- 指标字段回答“有多少”
- 状态字段回答“现在算什么状态”

如果把状态藏在金额、数量、日期差、利润差里，系统一开始可能能跑，但后面最容易在筛选、提醒、看板和工作流上出问题。

## 常见反模式

### 1. 用金额比较表达业务状态

例如：

- `<自动未结清金额字段> > 0` 代表未结清
- `利润 != 0` 代表有效订单
- `费用 > 0` 代表已入账

问题：

- 业务语义不清楚
- 遇到部分收款、冲销、赠送、补差价就会歪
- 图表和工作流不一定稳定支持复杂数字筛选

更稳的做法：

- `结算状态`
- `是否已结清`
- `是否已入账`
- `订单有效性`

### 2. 用数量差值表达流程状态

例如：

- `发货数量 < 订单数量` 代表待发货
- `退货数量 > 0` 代表售后中

问题：

- 适合计算，不适合表达完整流程
- 部分发货、已锁单、已取消时语义会混

更稳的做法：

- `发货状态 = 待发货 / 部分发货 / 已发货 / 已取消`
- `售后状态 = 无 / 处理中 / 已完成`

### 3. 用日期差表达当前状态

例如：

- `今天 - 下单日期 > 30` 代表沉默客户
- `今天 > 计划结算日期` 代表逾期

问题：

- 这是分析条件，不是完整状态
- 一旦要做提醒、分层、例外处理，就会反复复制判断逻辑

更稳的做法：

- `客户活跃状态`
- `结算状态 = 未到期 / 已到期 / 逾期 / 已结清`

### 4. 用文本快照代替结构化字段

例如：

- 订单里手填客户名称
- 订单里手填品牌名
- 看板读“客户文本快照”而不是关联客户

问题：

- 人工维护成本高
- 改名、合并、统计都会失真
- Agent 和用户录入时容易出现不同写法

更稳的做法：

- 主数据表 + 关联字段
- 自动公式字段读取主数据属性

### 5. 一个字段同时承担录入、状态、筛选、展示四种职责

例如：

- `备注` 既写业务说明，又写系统判断，又被拿去筛异常
- `状态` 既给用户看，又被工作流偷偷当控制开关

问题：

- 后续任何修改都容易牵一发动全身
- 用户看不懂，Agent 也难稳定使用

更稳的做法：

- 用户可读字段和系统控制字段拆开
- 备注字段只写用户明确表达的备注
- 工作流控制字段单独存在，并尽量隐藏

## 什么情况下应该补显式状态字段

满足任一条件，就应该优先考虑补字段，而不是继续堆公式判断：

- 同一个判断要在图表、视图、工作流里重复出现
- 用户会用这个状态直接筛选或看板展示
- 这个状态存在“部分完成”或多阶段过程
- 这个状态一旦判断错，会影响经营口径或后续动作
- 你已经开始写“金额大于 0 但不等于某些例外”的复杂条件

## 推荐的字段分层

一套稳定系统里，通常至少要有这几层：

### 1. 事实输入字段

用户真实录入：

- 数量
- 金额
- 日期
- 客户
- 商品
- 凭证

### 2. 自动指标字段

用于计算：

- 自动应收金额
- 自动未结清金额
- 自动利润
- 自动月份

### 3. 显式状态字段

用于筛选、提醒、图表、工作流：

- 是否已结清
- 结算状态
- 订单状态
- 是否本月
- 是否仍未结清

### 4. 展示字段

用于界面可读性：

- 客户显示名
- 品牌显示名
- 城市分组标签

## 最小修正策略

如果系统已经在线，不要为了补状态字段大改全表。优先用最小修法：

1. 保留原有金额和数量事实字段
2. 保留原有自动指标字段
3. 新增一个显式状态字段
4. 把图表、视图、工作流的筛选从复杂数值判断迁过去

这样通常能在不破坏历史数据的前提下，把系统稳定性拉起来。

## 和看板问题的关系

很多看板问题表面上是组件报错，本质上是字段分工错了。

典型信号：

- 图表值能算出来，但筛选一加就坏
- 同一逻辑在多个组件里重复写一遍
- 改一个业务口径，要改 5 个公式和 4 个筛选

这时不要只修图表，要回头看字段建模是不是把“状态”错塞进了“指标比较”里。
