# 现有系统维修流程

## 何时读取

当用户已有飞书 Base、表格、看板或工作流，并反馈“之前能用，现在不行”“图表报错”“筛选不动”“公式错了”“字段写不进去”时，读取这份 reference。

维修路径独立于搭建路径。不要重新问完整业务，也不要默认重建整套系统。

## 第一判断

先把问题归到一个最小对象：

| 问题类型 | 常见说法 | 先查什么 |
| --- | --- | --- |
| 字段问题 | 字段不见了、类型变了、写不进去 | 表结构、字段类型、字段可写性 |
| 公式问题 | 金额算错、利润不对、月份不对 | 公式字段、输入字段、样例记录 |
| lookup / 关联问题 | 关联不到、引用为空、跨表字段不对 | 关联字段、目标表、lookup 配置 |
| 看板问题 | 图表空白、配置变更、筛选不动 | dashboard、block、数据源、筛选字段 |
| 工作流问题 | 没触发、没反写、没提醒 | workflow 状态、触发条件、动作字段 |
| 数据导入问题 | 导入后格式错、字段错乱 | 导入映射、字段类型、样例数据 |
| 权限 / 配置问题 | 看不到、改不了、执行失败 | Base 权限、角色、当前账号 |

## 最小排障顺序

1. 确认出问题的是哪一个表、字段、视图、看板组件、工作流或记录。
2. 确认问题是显示不对、算错、写不进去、没触发，还是组件报错。
3. 确认之前是否正常，最近改过字段、公式、数据源、视图或权限没有。
4. 只读读取现状，不直接改。
5. 用一条或几条样例记录复现问题。
6. 拆出根因后做最小修复。
7. 修复后重新验证样例数据和页面渲染。

只有局部排障解释不了问题时，才扩大到整套系统。

## 看板和筛选问题

遇到看板、图表、筛选器、排序、组件空白或 `配置数据发生变更，请重新配置`，按顺序读取：

1. `dashboard-troubleshooting.md`
2. `dashboard-filter-slicer-case.md`
3. `field-modeling-antipatterns.md`

核心判断：

- 字段能聚合，不等于能筛选。
- 字段同名，不等于同一个数据源。
- 筛选器不一定能稳定控制跨表组件。
- `dashboard-block-update` 可能保留旧配置残留。
- 复杂公式字段、lookup 字段、关联展开字段做筛选前必须实测。

优先修法：

1. 保留公式指标，用显式状态字段或单选字段做筛选。
2. 把核心看板组件统一到同一事实表。
3. 如果旧 block 配置脏了，重建组件，不反复 patch。
4. 最后才考虑新增中间表或缓存表。

## 字段与公式问题

字段和公式问题先问：

- 这个字段是用户事实输入，还是自动派生字段？
- 这个字段是否被多个看板、视图、工作流引用？
- 这个字段是否把状态藏在金额、数量、日期差里？
- 这个公式是否依赖被改名、改类型或删除的字段？

如果一个状态要被筛选、提醒、看板、工作流反复使用，应补显式状态字段，不要继续堆复杂数值判断。

## 工作流问题

工作流只负责动作，不负责计算。适合检查：

- 是否启用。
- 触发条件是否仍然命中。
- 动作字段是否存在且可写。
- 是否依赖公式或 lookup 只读字段。
- 是否需要创建汇总行、反写状态、发送提醒。

不要用工作流维护客户名、商品名、品牌名快照，也不要用工作流做看板常规刷新。

## 修复输出

维修完成后输出：

1. 根因是什么。
2. 改了哪些表、字段、组件或工作流。
3. 用哪条样例数据验证通过。
4. 是否还有底层模型风险。
5. 是否清理了临时字段、组件、视图。

如果没有修改，只做定位，也要说明当前证据指向哪里，以及下一步需要用户提供什么。
