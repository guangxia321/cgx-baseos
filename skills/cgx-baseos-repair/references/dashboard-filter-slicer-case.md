# 看板筛选器不生效复盘

## 何时读取

遇到下面问题时，先读这份复盘，再动看板：

- 用户说“点了筛选，但数字不变”
- 月份、状态、客户、区域筛选看起来能选，但指标卡、排行、趋势不联动
- 看板同时读取事实表、汇总表、客户表或商品表
- 组件配置看似正确，但页面刷新后仍然不按筛选变化
- 用户想把文本月份改成单选月份，方便直接点选

这类问题通常很耗时间，因为表面像是某个字段坏了，本质常常是“数据源、筛选器、字段类型、前端联动状态”四件事叠在一起。

## 脱敏复盘摘要

一个经营系统的月度看板原本有多个指标卡和排行组件。用户发现不管怎么选择月份，销售额、订单数、未收款、品牌排行、商品排行都不变化。

初始判断以为是“本月”公式或月份字段有问题，但后续验证发现真正的关键点是：

- 看板组件混用了两类数据源：订单事实表和月度汇总表。
- 月份筛选器实际只作用在其中一张表上。
- 即使两张表里都有一个叫“月份”的字段，飞书看板也不会自动认为它们是同一个筛选维度。
- 后来把组件迁到同一张订单事实表后，指标开始能被同一个月份筛选器控制。
- 迁移后还发现部分公式字段能显示，但聚合结果为空，需要把数值公式显式转成数字。
- 最后，筛选器外观看起来已经切到新字段，但内部联动规则仍残留旧表字段，必须检查页面运行态或重建筛选器。

最终解决路径不是重建整套系统，而是：

1. 给订单事实表增加显式单选月份字段。
2. 历史订单按自动月份回填这个单选字段。
3. 录入链路同步写入这个单选月份字段。
4. 将月度看板核心组件统一到订单事实表。
5. 指标字段仍使用自动公式，但用显式单选月份做筛选维度。
6. 重配或修正筛选器联动规则，确保它指向订单事实表的单选月份字段。
7. 完整刷新页面后，验证两个不同月份切换时，指标卡、趋势、品牌排行、商品排行都发生变化。

## 关键根因

### 1. 字段同名不等于筛选同源

飞书看板中，一个组件读“订单事实表.月份”，另一个组件读“月度汇总表.月份”，即使两个字段显示名都叫“月份”，筛选器也不一定能同时控制它们。

排障时不要只看字段名，要确认：

- 筛选器绑定的是哪张表
- 每个组件的数据源是哪张表
- 组件筛选条件里的 fieldId 属于哪张表
- 筛选器运行态的联动规则是否还残留旧表字段

### 2. 多数据源筛选可能不是可用能力

某些飞书环境里，多数据源切片器或跨表筛选可能受套餐、前端能力或组件类型限制。不能假设“一个筛选器控制所有表”天然可用。

如果当前环境无法稳定支持多数据源筛选，优先策略是：

- 把当前看板核心指标统一到同一张事实表
- 用同一个显式筛选字段控制这些组件
- 汇总表继续作为后台校验或其他视图使用，而不是让用户手工维护

### 3. 月份筛选应建成显式维度

月份如果只是公式文本，用户在看板筛选里常常需要手动输入，体验差，也容易和选项值、显示值、内部 ID 混淆。

更稳的设计是：

- 保留 `自动月份` 作为公式或自动结果
- 新增 `月份筛选` 作为单选字段
- 由脚本、工作流或批量修正自动写入 `月份筛选`
- 看板用 `月份筛选` 做筛选和分组

这类字段是展示和筛选维度，不是让用户额外维护的事实字段。

### 4. 公式数字要验证能否聚合

有些自动金额、利润、数量字段在表格里看起来是数字，但图表聚合时可能返回空或异常。常见原因是公式结果被前端或图表层识别成非纯数字。

修法通常是把用于图表的数值公式显式包成数字结果，例如：

```text
VALUE(IFERROR(<原公式>, 0))
```

注意先在少量字段上验证，不要一次性大改全部公式。

### 5. 筛选器配置和运行态都要查

看板组件的服务端 `data_config` 正确，不代表页面里筛选器的联动规则也正确。

特别是曾经改过数据源、复制过组件、尝试过多数据源筛选时，可能出现：

- 筛选器数据源已显示成新表
- 但联动规则仍指向旧表字段
- 页面上选择了月份，组件却不动

这时要么重建筛选器，要么确认并修正筛选器的联动对象。

## 标准排障流程

### 第一步：列出所有组件和数据源

对目标看板先拿到：

- 组件列表
- 每个组件类型
- 每个组件的 `table_name`
- `series`
- `group_by`
- `filter`

目标是先回答一个问题：这个看板到底读了几张表？

如果同一个筛选要控制多个组件，但这些组件来自不同表，先不要继续调字段，要先处理数据源联动问题。

### 第二步：确认筛选字段类型

月份、状态、区域、客户分层这类筛选条件，优先使用：

- 单选
- 多选
- checkbox
- 明确状态字段
- 稳定日期标签字段

不优先使用：

- 手填文本
- 复杂公式文本
- lookup 结果
- 关联展开字段
- 复杂公式数字比较

### 第三步：创建或修正显式筛选维度

如果用户想“直接选月份”，通常应该做：

- 新增 `月份筛选` 单选字段
- 选项格式固定，例如 `YYYY/MM`
- 历史记录回填
- 新增记录时自动写入
- 看板筛选器绑定这个字段

不要让用户在看板筛选框里手动输入月份。

### 第四步：统一核心看板数据源

对于月度经营看板，如果同一个月份筛选要控制销售额、订单数、桶数、利润、未收款、品牌排行、商品排行，优先把这些组件统一到订单事实表：

- 销售额：订单事实表的自动应收金额求和
- 订单数：订单事实表的订单计数字段求和，或用按订单号去重后的可聚合字段
- 数量：订单事实表的自动换算数量求和
- 利润：订单事实表的自动利润字段求和
- 未收款：订单事实表的自动待收金额求和
- 品牌排行：订单事实表按自动品牌分组
- 商品排行：订单事实表按商品分组

如果经营费用等数据来自财务流水，可以用分摊公式、汇总字段或单独组件处理，但必须单独验证它是否能被同一个月份筛选稳定控制。

### 第五步：处理订单数这类去重指标

一单多明细时，直接统计行数会把订单数放大。可用一个可聚合的订单计数字段：

```text
1 / 同订单号明细数量
```

再在看板里对这个字段求和。这样同一个订单有多行明细时，加起来仍然是 1。

### 第六步：验证筛选器联动规则

不要只看页面上有一个筛选框。要确认：

- 筛选器数据源是目标事实表
- 筛选器分组字段是目标单选字段
- 筛选器联动规则里的 tableId 是目标事实表
- 条件字段是目标单选字段
- 顶部全局筛选没有残留旧条件

如果确认不了内部联动规则，直接删掉旧筛选器，用已验证的字段重新建一个。

### 第七步：完整重载页面后做双月份验证

必须至少选两个不同月份验证：

- 指标卡数值变化
- 趋势图变化
- 品牌排行变化
- 商品排行变化
- 未收款或经营利润等关键口径变化

只验证“当前月份有数”不够。必须验证从 A 月切到 B 月后，所有相关组件都跟着变。

## 收款排行的相关教训

“未收款客户排行”这类组件经常会踩另一个坑：用 `自动待收金额 > 0` 做图表筛选。

自动待收金额作为指标求和可能正常，但作为数字筛选条件可能触发图表兼容问题，返回空数据或报错。

更稳的方案：

- 指标仍然用 `自动待收金额` 求和
- 筛选先用明确状态字段，例如 `是否收款 isNot 已收款`
- 如果存在部分收款，长期应补一个显式状态：`结算状态 = 未结清 / 部分结清 / 已结清`

也就是说，金额字段回答“欠多少”，状态字段回答“算不算未结清”。不要把这两件事混在一个公式比较里。

## 不要做的事

- 不要一开始就新建看板缓存表。
- 不要要求用户手工维护汇总表。
- 不要只因为字段名相同，就认为跨表筛选会生效。
- 不要只通过 `dashboard-block-update` 成功就认为页面已修好。
- 不要在没有验证前，把复杂公式数字字段用作图表筛选条件。
- 不要保留排查时创建的临时组件、临时字段、临时视图。

## 验收标准

这类问题只有同时满足下面条件才算修好：

- 服务端组件配置读取出来是正确的
- 页面完整重载后组件仍正常渲染
- 筛选器默认值正确或无残留旧筛选
- 至少两个筛选值切换后，所有目标组件数值都变化
- 新增数据链路会自动写入筛选维度
- 临时测试组件已清理
- 用户无需手工维护汇总表或缓存表

## 抽象原则

这次问题的通用结论是：

看板筛选不生效时，不要只盯着筛选框。先把“数据源是否一致、筛选字段是否稳定、公式字段是否能聚合、筛选器联动规则是否残留旧表”四件事拆开。

最快的稳定方案通常不是重建系统，而是把用户要筛选的核心看板统一到同一张事实表，并给筛选维度补一个显式、可选、自动维护的结构化字段。
