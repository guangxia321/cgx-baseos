# 广厦 BaseOS Build

用于把业务数据地图落成可长期使用的经营数仓和飞书 Base。先确认业务数据地图，再按四阶段推进；没有地图时，先切回 `$cgx-baseos-diagnose` 做诊断。

## 必读顺序

1. `references/build-playbook.md`：四阶段构建、重构、模板、飞书落地和验收流程。
2. `references/system-design-rules.md`：基础数据、录入联动、字段分层、工作流边界。
3. `references/dashboard-build-rules.md`：新建看板设计规则和上线验收。
4. `references/high-frequency-scenarios.md`：确认第一版主链路和模板映射。
5. 如果缺少一页业务数据地图：读 `references/business-classifier.md`，并切回 `$cgx-baseos-diagnose`。

## 四阶段运行协议

### 阶段 1：基础数据层

- 确认主数据、事实数据、历史数据、派生数据和展示数据。
- 确认字段类型、关联关系、历史数据迁移方式和哪些旧结构可保留。
- 输出基础表结构，不设计看板组件。

### 阶段 2：录入与联动层

- 设计日常录入页、批量录入页、Agent 共用入口和导入校验视图。
- 明确用户录入字段、自动字段、lookup/关联字段、显式状态字段。
- 设计必要公式和工作流；公式算指标，工作流只做动作。

### 阶段 3：展示看板层

- 只有基础数据层和录入联动层稳定后，才进入看板。
- 明确每个指标的数据源、指标字段、筛选字段、分组字段和组件类型。
- 看板优先读取同一事实表或稳定汇总表；筛选字段优先用单选、状态字段、月份筛选字段。
- 不默认新增看板缓存表。

### 阶段 4：验收发布层

- 用 3-5 条样例数据验证录入、公式、lookup、状态、工作流和看板。
- 看板必须完整重载，并用至少两个筛选值切换验证。
- 确认用户日常使用路径、批量导入路径、Agent 查询/录入路径和旧系统未误改。

## 按需读取

- 涉及收款、应收、费用、利润、现金流、凭证或项目利润：读 `references/finance-scenarios.md`。
- 用户只要字段清单、Excel 模板或飞书表结构：读对应 `templates/*/schema.md`，但不要切到独立模板技能。
- 用户提到官方模板、行业模板、成熟案例、相同/差异点：读 `references/official-template-design-patterns.md` 和 `references/official-template-reference-library.md`。
- 用户问公式、数据流转、状态处理、工作流边界、看板设计：读 `references/official-template-execution-patterns.md`。
- 方案确认且要落到飞书 Base：读 `references/feishu-cli-setup.md`，先完成 CLI、授权、目标 Base 和权限验证，再使用 `lark-base` skill。
- 已有系统出错、看板坏了、字段/公式/工作流异常：切到 `$cgx-baseos-repair`。

## 执行规则

- 第一版只做能长期使用的最小系统。
- 不跳阶段：不能在基础数据和录入联动未稳定时直接设计看板。
- 必须设计日常录入视图、批量录入视图、Agent 入口和导入校验视图。
- 用户和 Agent 要共用同一套数据入口。
- 财务结构只做经营可见性，不替代专业会计、报税、总账或复式记账。
- 用户要求落到飞书 Base 时，CLI、授权、目标 Base 和修改权限是硬前置；未通过前只能给方案、模板或本地验算结果，不能改用本地 Excel 作为默认交付。
- 默认不做复杂权限、复杂审批、完整库存、完整 ERP、人事、绩效。

## 输出

方案阶段按四阶段输出：基础数据层、录入与联动层、展示看板层、验收发布层。每阶段都要说明已确认内容、待确认问题和不做的内容。

落地完成后输出：建了或改了哪些表、字段、视图、看板、工作流；用户日常怎么用；哪些字段不用填；样例数据和关键口径是否核对；看板是否通过双筛选值验证；还有哪些明确待办。
