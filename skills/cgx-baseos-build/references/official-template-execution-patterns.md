# 飞书官方模板执行层模式手册

## 何时读取

当用户的问题涉及公式、字段分层、数据轮转、状态处理、lookup/rollup、工作流边界、看板设计，或问“这个系统具体怎么搭才稳”时，读取这份 reference。

这份资料和 `official-template-design-patterns.md` 分工不同：

- `official-template-design-patterns.md` 负责判断“应该有哪些对象、表和业务闭环”。
- 本文件负责判断“字段怎么分层、公式怎么写、数据怎么流、看板怎么稳、工作流什么时候用”。

## 1. 执行层共通模式

官方模板真正值得吸收的执行经验，是把字段和动作分层，而不是把所有信息塞进一张表。

| 层级 | 典型字段 | 谁维护 | 作用 |
| --- | --- | --- | --- |
| 用户录入字段 | 客户、商品、数量、金额、日期、负责人、附件、备注 | 用户或 Agent | 记录真实发生的事实 |
| 标准化字段 | 自动编号、自动月份、分类、阶段、状态、时间标签 | 系统自动生成或少量选择 | 统一口径，减少手填差异 |
| 关联/lookup 字段 | 客户负责人、产品价格、合同金额、项目客户、库存商品 | 系统读取 | 从主数据和关联表继承信息 |
| 自动公式字段 | 应收、实收、待收、利润、完成率、延期天数、当前库存 | 公式 | 计算指标，不承载人工判断 |
| 显式状态字段 | 结算状态、发货状态、履约状态、工单状态、是否逾期 | 单选/复选/公式 | 给筛选、视图、看板、工作流使用 |
| 展示字段 | 组合标题、客户显示名、月份筛选、看板分组标签 | 公式或单选 | 提高可读性和看板稳定性 |

默认原则：用户只填事实，系统补口径，公式算指标，状态服务筛选，工作流只做动作。

## 2. 录入入口设计

所有经营系统都有一个核心动作：录入。系统能不能长期用，首先取决于录入入口是否足够方便、足够标准化。

设计录入入口前，先诊断：

1. 用户平时最花时间录入的是什么。
2. 哪个动作最没有标准化，最容易不同人填出不同格式。
3. 哪些字段是每次都必须填的事实，哪些可以从主数据、公式、lookup 或工作流得到。
4. 录入是多人各自填，还是经常从 Excel/表格复制后批量粘贴。
5. 录入后马上要触发什么：计算、状态更新、提醒、看板、汇总、归档。

### 两类录入场景

| 场景 | 适合入口 | 设计重点 |
| --- | --- | --- |
| 成员日常录入 | 飞书 Base 表单视图、表格视图、移动端友好视图、个人待办视图 | 字段少、选项清楚、默认值明确、只展示该角色要填的字段 |
| Excel/表格批量录入 | 可粘贴的标准导入表、批量录入视图、字段顺序固定的事实表 | 列顺序稳定、字段类型容错、必填字段清楚、导入后自动标准化 |

### 日常录入入口

成员各自录入时，入口要追求“少而准”：

- 只展示用户需要填的事实字段，不展示后台公式、lookup 和汇总字段。
- 客户、商品、项目、负责人这类字段优先用关联或人员字段，不让用户手打文本。
- 状态字段使用单选、复选或固定选项，不让用户在备注里写状态。
- 日期、月份、编号尽量自动生成；确实要用户确认的日期才展示。
- 不同角色可以有不同视图，例如销售录客户和订单，财务录收款和发票，交付录任务和进度。
- Agent 代录时也走同一套事实入口，不依赖隐藏脚本或另一张内部表。

### 批量录入入口

从 Excel 或旧表复制到飞书 Base 是高频场景，应该把它当成正式入口设计，而不是临时导入。

批量录入入口的设计规则：

- 提供一张字段顺序稳定的事实表或导入视图，列顺序尽量贴近用户现有 Excel。
- 必填列放前面，自动字段放后面并隐藏或锁定。
- 允许用户批量粘贴事实字段，但不要求用户粘贴公式字段、lookup 字段和汇总字段。
- 如果历史数据里客户名、商品名是文本，先设计匹配规则，再决定是否批量关联到主数据。
- 导入后必须有校验视图：缺客户、缺商品、金额为空、日期异常、状态未识别、重复订单号。
- 批量导入不等于绕过标准化。导入只是录入手段，最终仍应沉淀成主数据、事实表和自动字段。

### 录入入口的字段顺序

事实表的字段顺序建议：

1. 人能识别的核心字段：订单号、项目名、客户、商品、工单标题。
2. 用户必须填的事实字段：日期、数量、金额、负责人、附件。
3. 选择型标准字段：状态、阶段、分类、渠道、来源。
4. 备注和补充说明。
5. 系统自动字段：编号、月份、lookup、公式、汇总、展示标签。

这样既适合日常表格录入，也适合从 Excel 按列复制粘贴。

### 录入后的自动处理

一次录入后，系统应该自动完成这些事：

- 自动补编号、月份、负责人继承、客户/商品属性 lookup。
- 自动计算金额、利润、待收、完成率、库存余额。
- 自动更新显式状态，例如结算状态、履约状态、库存预警状态。
- 必要时触发工作流，例如创建汇总行、提醒负责人、反写状态。
- 看板无需用户刷新，能直接读取事实字段和自动字段。

录入入口的验收标准：用户新增一条事实记录后，不需要再去别的表补同一件事的派生信息。

## 3. 数据流转路径

稳定系统通常按这条路径流转：

```text
主数据 -> 事实表 -> 过程记录 -> 财务/数量明细 -> 派生汇总 -> 看板/提醒
```

各层职责：

- 主数据：客户、产品、服务、供应商、仓库、人员、门店、平台。负责稳定对象。
- 事实表：订单、合同、项目、任务、工单、直播场次、案件。负责业务发生。
- 过程记录：跟进、日报、回访、处理记录、复盘。负责历史过程。
- 财务/数量明细：收款、付款、发票、报销、入库、出库、退货、调拨。负责核算依据。
- 派生汇总：订单汇总、月度汇总、实时库存、目标完成、项目回款。负责自动聚合。
- 看板/提醒：销售额、未收款、进度、库存、满意度、逾期。负责行动判断。

不要让上游依赖下游。例如：订单不应该依赖看板缓存；收款状态不应该靠看板筛选判断；库存余额不应该由用户在看板里手工修。

## 4. 工具职责边界

| 能力 | 适合做 | 不适合做 |
| --- | --- | --- |
| 关联记录 | 表达客户-订单、订单-收款、项目-任务、产品-库存 | 替代状态字段 |
| lookup / rollup | 读取主数据属性、聚合收款金额、汇总任务数量 | 作为复杂筛选的唯一依据 |
| 公式 | 金额、利润、月份、延期、完成率、库存余额 | 创建记录、发送提醒、处理人工例外 |
| 显式状态字段 | 筛选、看板、提醒、工作流条件 | 代替金额和数量事实 |
| 工作流 | 自动补行、反写状态、提醒、通知、归档 | 算金额、刷新看板、维护文本快照 |
| 汇总表 | 跨记录聚合、月度/订单/库存台账 | 用户手填经营结果 |
| 看板 | 展示指标、趋势、排行、异常 | 成为系统数据源 |

判断公式和工作流的边界：能用公式稳定表达的，就不要交给工作流；需要创建、更新、通知、反写的，才交给工作流。

## 5. Pattern Card：销售订单回款

### 适用场景

用户要管理客户、订单、合同、开票、收款、未收款和销售看板。

### 推荐表流转

```text
客户 -> 联系人/商机 -> 合同/订单 -> 收款记录/发票 -> 订单汇总/月度汇总 -> 销售看板
```

轻量业务可以跳过联系人、商机和合同，把主链路压缩成：

```text
客户 -> 订单 -> 收款记录 -> 销售看板
```

### 必备字段分层

| 分层 | 字段 |
| --- | --- |
| 用户录入字段 | 客户、订单日期、商品/服务、数量、成交金额、收款日期、收款金额、凭证 |
| 关联/lookup 字段 | 客户负责人、客户来源、商品价格、合同金额、发票信息 |
| 自动公式字段 | 应收金额、已收金额、待收金额、自动月份、利润、订单计数字段 |
| 显式状态字段 | 结算状态、开票状态、订单状态、是否仍未结清 |
| 展示字段 | 订单标题、客户显示名、月份筛选 |

### 常见公式/状态口径

- `待收金额 = 应收金额 - 已收金额`。
- `结算状态 = 未收款 / 部分收款 / 已结清 / 已取消`。
- 一单多商品时，订单数不要直接数行，可用可聚合订单计数字段。
- 月份既可以保留自动公式，也要给看板准备稳定的 `月份筛选` 单选字段。

### 看板组件设计

- 指标卡：销售额、已收款、待收款、订单数、利润。
- 趋势：按月份看销售额和回款。
- 排行：客户排行、商品排行、销售负责人排行。
- 异常：未结清客户、逾期收款、未开票订单。

看板指标优先读订单事实表和收款事实表；筛选优先用月份筛选、结算状态、销售负责人。

### 什么时候需要工作流

- 新订单后补订单汇总行。
- 新收款后反写订单已收金额或触发状态更新。
- 到期未收款提醒负责人。
- 合同到期或开票缺失提醒。

### 不建议照抄的地方

- 不要只用 `待收金额 > 0` 作为未收款筛选。
- 不要在订单里手填客户名、产品名、销售负责人快照。
- 不要让用户手工维护月度收入统计。

### 验收方式

- 新增订单后，应收、月份、客户/商品 lookup 自动出现。
- 新增收款后，已收、待收、结算状态联动。
- 切换两个不同月份后，销售额、订单数、未收款、排行都变化。

## 6. Pattern Card：CRM 商机跟进

### 适用场景

用户销售周期较长，需要管理线索、客户、联系人、商机阶段、跟进记录和预测成交。

### 推荐表流转

```text
线索 -> 客户/联系人 -> 商机 -> 跟进记录 -> 合同/订单 -> 销售目标/看板
```

### 必备字段分层

| 分层 | 字段 |
| --- | --- |
| 用户录入字段 | 线索来源、客户、联系人、商机名称、预计金额、预计成交日期、跟进内容 |
| 关联/lookup 字段 | 客户行业、客户负责人、联系人电话、关联合同 |
| 自动公式字段 | 商机年龄、预计月份、最近跟进间隔、目标完成率 |
| 显式状态字段 | 线索状态、商机阶段、输赢状态、是否超期未跟进 |
| 展示字段 | 商机标题、客户分层、销售漏斗阶段 |

### 常见公式/状态口径

- 商机阶段用于漏斗，不要用备注描述阶段。
- 最近跟进时间可以 lookup 最新跟进记录，超期状态再由公式或工作流维护。
- 输单原因单独结构化，不要埋在跟进备注里。

### 看板组件设计

- 销售漏斗：按商机阶段统计数量和预计金额。
- 跟进看板：超期未跟进、今日待跟进、销售负责人维度。
- 成交预测：预计成交月份、预计金额、赢单率。

### 什么时候需要工作流

- 新线索达到条件后转客户。
- 商机长时间未跟进提醒。
- 商机赢单后创建合同/订单草稿。

### 不建议照抄的地方

- 轻量业务不要强行上线线索、商机、联系人全套。
- 不要把商机阶段和订单状态混在一个字段。

### 验收方式

- 一个客户能看到联系人、商机、跟进、合同。
- 商机阶段变化后漏斗可统计。
- 跟进记录新增后最近跟进时间和超期状态能更新。

## 7. Pattern Card：项目交付经营

### 适用场景

用户要管理项目、任务、交付进度、预算、成本、回款和项目经营分析。

### 推荐表流转

```text
客户/合同 -> 项目 -> 任务/日报/周报 -> 预算/报销/回款 -> 项目汇总 -> 项目看板
```

### 必备字段分层

| 分层 | 字段 |
| --- | --- |
| 用户录入字段 | 项目名称、负责人、计划开始/完成、任务、实际完成、日报、报销金额 |
| 关联/lookup 字段 | 客户、合同金额、项目成员、任务数量、已回款 |
| 自动公式字段 | 完成率、延期天数、剩余预算、项目毛利、回款进度 |
| 显式状态字段 | 项目阶段、任务状态、是否逾期、风险等级 |
| 展示字段 | 项目标题、周报摘要、进度标签 |

### 常见公式/状态口径

- `是否逾期` 不只看日期，还要排除已完成/已取消。
- 完成率可以来自任务状态汇总，不建议由用户手填。
- 项目回款和项目成本要能追溯到明细，不只在项目表填总数。

### 看板组件设计

- 项目总览：进行中、逾期、高风险、待回款。
- 进度看板：按阶段、负责人、计划完成日期。
- 经营看板：合同金额、已回款、成本、毛利。

### 什么时候需要工作流

- 新项目后自动生成标准任务清单。
- 任务逾期提醒负责人。
- 周报提交后通知项目经理。
- 回款到期提醒。

### 不建议照抄的地方

- 不要把所有任务进展写在项目备注里。
- 不要只做任务表而没有项目经营口径。
- 不要让用户手填完成率。

### 验收方式

- 项目能关联任务、预算、报销、回款。
- 任务状态变化后项目完成率能变化。
- 项目看板能同时看到进度风险和经营结果。

## 8. Pattern Card：进销存库存

### 适用场景

用户有实物商品，需要管理采购、销售、入库、出库、退货、调拨、盘点和实时库存。

### 推荐表流转

```text
产品/供应商/仓库 -> 采购订单 -> 采购入库/付款
产品/客户/仓库 -> 销售订单 -> 销售出库/收款
库存流水 -> 实时库存 -> 库存看板
```

### 必备字段分层

| 分层 | 字段 |
| --- | --- |
| 用户录入字段 | 产品、仓库、数量、单价、供应商、客户、入库/出库日期 |
| 关联/lookup 字段 | 产品编码、规格、供应商信息、仓库位置、订单数量 |
| 自动公式字段 | 采购金额、销售金额、已入库数量、已出库数量、当前库存、库存预警 |
| 显式状态字段 | 入库状态、出库状态、付款状态、回款状态、库存预警状态 |
| 展示字段 | 产品库存标题、仓库分组、月份筛选 |

### 常见公式/状态口径

- 实时库存要按产品 + 仓库维度计算。
- 入库、出库、退货、调拨、盘点应作为数量明细，不要只改产品表库存数。
- 库存预警状态用显式字段展示，公式计算库存余额。

### 看板组件设计

- 库存总览：当前库存、低库存、高库存、库存金额。
- 采购看板：待入库、待付款、供应商维度。
- 销售看板：待出库、待回款、客户维度。
- 异常看板：负库存、超安全库存、长期未动销。

### 什么时候需要工作流

- 审核通过后生成入库/出库待办。
- 库存低于安全线提醒采购。
- 盘点差异超过阈值提醒负责人。

### 不建议照抄的地方

- 没有真实库存压力时，不要上完整进销存。
- 不要让用户直接维护 `当前库存`。
- 不要把采购、销售、退货、调拨混在一个无类型流水里，除非类型字段非常稳定。

### 验收方式

- 同一产品在不同仓库的库存能分开计算。
- 采购入库、销售出库、退货、调拨会影响实时库存。
- 库存看板能筛选仓库、产品分类和预警状态。

## 9. Pattern Card：工单售后服务

### 适用场景

用户要管理售后、投诉、客户问题、内部支持、处理进度和满意度。

### 推荐表流转

```text
客户/产品 -> 工单 -> 处理记录/回访 -> 满意度 -> 工单看板
```

### 必备字段分层

| 分层 | 字段 |
| --- | --- |
| 用户录入字段 | 客户、产品、问题描述、分类、紧急程度、期望完成日期、附件 |
| 关联/lookup 字段 | 客户负责人、产品信息、历史工单、处理人 |
| 自动公式字段 | 工单时长、逾期天数、响应时长、满意度均值 |
| 显式状态字段 | 工单状态、处理状态、是否逾期、满意度等级 |
| 展示字段 | 工单编号、问题标题、客户显示名 |

### 常见公式/状态口径

- 工单状态应覆盖待处理、处理中、待回访、已完成、已关闭。
- 紧急程度和逾期状态是两件事，不要混成一个字段。
- 满意度记录应独立保存，不要覆盖工单备注。

### 看板组件设计

- 工单总览：待处理、处理中、逾期、已关闭。
- 负责人看板：每人未完成工单和平均处理时长。
- 质量看板：满意度、投诉类型、复发问题。

### 什么时候需要工作流

- 新工单通知负责人。
- 超过响应时限提醒。
- 工单完成后自动发起回访。
- 低满意度自动标记复盘。

### 不建议照抄的地方

- 不要只在客户表加一个售后备注字段。
- 不要把满意度当成工单状态。
- 不要用创建时间差直接替代完整 SLA 状态。

### 验收方式

- 工单能看到客户、产品、处理记录、满意度。
- 状态变化后看板指标能变化。
- 逾期工单能被视图、看板和提醒同时识别。

## 10. Pattern Card：电商直播运营

### 适用场景

用户要管理商品/SKU、平台、达人、直播场次、销售明细、退货、样品库存和复盘。

### 推荐表流转

```text
商品/SKU + 平台 + 达人 -> 直播场次 -> 销售明细/退货 -> 样品出入库 -> 直播复盘/经营看板
```

### 必备字段分层

| 分层 | 字段 |
| --- | --- |
| 用户录入字段 | 直播场次、商品、平台、达人、订单金额、退货金额、样品出入库数量 |
| 关联/lookup 字段 | 商品成本、商品品牌、达人状态、平台类型 |
| 自动公式字段 | GMV、净销售额、毛利、退货率、样品库存、场次月份 |
| 显式状态字段 | 合作状态、样品状态、直播状态、复盘状态 |
| 展示字段 | 场次标题、商品显示名、月份筛选 |

### 常见公式/状态口径

- GMV、退货、净销售额、毛利要分开，不要只保留最终销售额。
- 直播场次是运营事实，不要只按日期文本记在销售明细里。
- 样品库存是独立小库存，不要和正式销售库存混用，除非业务确实一致。

### 看板组件设计

- 场次看板：GMV、退货率、毛利、场次表现。
- 商品看板：商品/SKU 排行、品牌表现、毛利排行。
- 达人看板：达人合作状态、场次贡献、复盘结果。
- 样品看板：样品库存、出入库、待归还。

### 什么时候需要工作流

- 直播结束后提醒复盘。
- 样品借出到期提醒。
- 达人合作状态变化后通知运营。

### 不建议照抄的地方

- 不要为了直播模板强行加入内容排期、话术、AI 复盘，除非用户真的要管理这些动作。
- 不要把销售明细、直播场次、样品出入库混成一张流水表。

### 验收方式

- 一场直播能关联达人、平台、商品和销售明细。
- GMV、退货率、毛利能分开看。
- 场次、商品、达人三个维度的看板都能联动。

## 11. 看板稳定设计规则

看板设计顺序：

1. 先确定看板服务谁：老板、销售、交付、财务、运营、客服。
2. 再确定核心问题：销售多少、欠多少、进度如何、库存是否危险、客户是否满意。
3. 再确定数据源：优先事实表和自动字段。
4. 再确定筛选维度：月份、状态、负责人、客户、商品、仓库。
5. 最后选择组件：指标卡、趋势图、排行、分布、明细列表。

稳定规则：

- 公式字段优先做指标值，显式状态字段优先做筛选条件。
- 月份、状态、负责人这类用户常筛字段，优先做成单选、人员、checkbox 或明确日期标签。
- 同一个筛选器想控制多个组件时，核心组件优先统一到同一张事实表。
- 字段同名不代表同源；跨表筛选必须实测。
- 复杂公式、lookup、关联展开字段做筛选前必须实际验证。
- 页面完整重载后仍能渲染，才算看板配置稳定。

## 12. 执行层验收清单

搭建或重构完成后，至少检查：

- 用户日常视图只展示用户录入字段和必要状态字段。
- 批量录入视图字段顺序稳定，能从 Excel/旧表复制粘贴事实字段。
- 导入后有校验视图能发现缺客户、缺商品、金额为空、日期异常、状态未识别和重复编号。
- 自动公式字段不用用户手填，并能在样例数据上算对。
- 显式状态字段能被视图、看板、工作流同时使用。
- lookup 字段能从主数据或关联事实中正确读取。
- 新增一条核心事实记录后，相关汇总、状态和看板能联动。
- 看板指标和筛选字段分层清楚，不依赖复杂数字比较做关键筛选。
- 工作流只负责创建、更新、提醒、反写，不负责核心金额计算。
- 至少用两个不同月份、两个不同状态验证看板切换。
- 没有让用户手工维护汇总表、看板缓存表或统计结果。

## 13. 输出给用户时的固定结构

当用户要方案、模板或系统搭建建议时，输出中要固定区分：

- 用户录入字段：用户或 Agent 每天真正需要填什么。
- 录入入口设计：成员日常录入怎么填，Excel/表格批量录入怎么贴，导入后怎么校验。
- 自动公式字段：系统自动算什么，例如金额、利润、月份、完成率。
- 显式状态字段：哪些状态用于筛选、提醒、看板和工作流。
- lookup/关联字段：哪些信息从主数据或其他表读取。
- 工作流动作：哪些动作需要自动创建、提醒、反写。
- 看板读取字段：每个核心看板指标读哪张表、哪个字段。

这比只给“表结构清单”更重要。执行层清楚，系统才会长期可维护。
