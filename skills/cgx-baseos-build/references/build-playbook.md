# 系统构建与重构流程

## 何时读取

当入口判断为构建系统、重构旧表、套用模板、经营财务结构、看板新建或飞书 Base 落地时，读取这份 reference。维修现有系统时优先读 `repair-playbook.md`，不要直接走完整构建流程。

涉及表结构、录入入口、订单字段、看板、工作流边界或验收细节时，同时读取 `system-design-rules.md`。涉及新建看板时读取 `dashboard-build-rules.md`。

## 构建前置判断

构建前必须先确认是否已有“一页业务数据地图”。

- 如果已有地图，基于地图进入构建。
- 如果没有地图，但用户只是要一个轻量模板或字段清单，可以先输出模板，同时提醒正式构建前仍需诊断。
- 如果没有地图且用户要搭真实系统，先切回 `$cgx-baseos-diagnose`，不要直接建表。

进入构建后：

1. 读取 `high-frequency-scenarios.md`，选择第一版主链路。
2. 读取 `system-design-rules.md`，套用经营系统通用设计规则。
3. 读取 `dashboard-build-rules.md`，明确展示看板层的设计和验收边界。
4. 如果用户提到官方模板、行业模板、成熟案例、相同/差异点或系统结构参考，读取 `official-template-design-patterns.md`。
5. 如果用户提到公式、数据流转、状态处理、工作流边界、看板设计或执行层经验，读取 `official-template-execution-patterns.md`。
6. 如果涉及收款、支出、利润、现金流或凭证，读取 `finance-scenarios.md`。
7. 如果要真正落到飞书 Base，读取 `feishu-cli-setup.md`，先完成 CLI 安装、配置、授权、目标 Base 和权限验证。

## 四阶段构建协议

### 阶段 1：基础数据层

目标是确定系统底板，不讨论看板组件。

必须输出：

- 主数据：客户、产品/服务、项目、合同、渠道、财务科目等稳定对象。
- 事实数据：订单、收款、支出、交付记录、跟进记录等真实发生的记录。
- 历史数据：旧表、Excel、SaaS、聊天记录或现有 Base 中可迁移的数据。
- 派生数据：订单汇总、月度汇总、利润、未收款等自动生成数据。
- 关联关系：哪些表关联，哪些字段 lookup，哪些字段只保留事实。
- 字段类型：文本、单选、数字、日期、人员、附件、关联、公式等。

已有旧表时：

1. 先只读扫描旧系统的表、字段、公式、lookup、视图、工作流、看板和历史记录。
2. 把旧数据映射到业务数据地图：哪些是主数据、事实数据、派生数据、展示数据。
3. 标出已经跑通的结构，优先保留。
4. 找出重复字段、手填快照、错误公式、断开的关联、无人维护的汇总表。
5. 设计新结构或局部改造方案，不直接覆盖旧系统。

### 阶段 2：录入与联动层

目标是让人和 Agent 能低成本、长期稳定地录入和联动。

必须输出：

- 主数据录入入口：客户、商品/服务、项目、门店、人员等主数据表要有登记/维护视图、可用目录视图；有历史清单时加批量导入视图，有外部或移动端提交时加表单入口。
- 日常录入页：给成员或 Agent 单条新增事实记录，只展示必要字段。
- 批量录入页：给 Excel/旧表复制粘贴，字段顺序固定，自动字段隐藏或放后面。
- Agent 共用入口：Agent 代录、查询、提醒时使用同一套事实入口。
- 导入校验视图：检查缺客户、缺商品、金额为空、日期异常、状态未识别、重复编号。
- 自动字段：编号、月份、金额、利润、待收、完成率等。
- 显式状态字段：结算状态、履约状态、任务状态、是否逾期、月份筛选等。
- 工作流动作：补汇总行、反写状态、发送提醒、补标准任务。

执行边界：

- 主数据的不同录入形态优先用同表不同视图或表单，不默认新增“客户录入表”“产品录入表”这类重复表。
- 只有产品上架申请、调价记录、样品入库、服务变更审批这类独立业务事实，才从主数据表拆出单独事实表。
- 用户只填事实，系统补口径，公式算指标，状态服务筛选，工作流只做动作。
- 不让用户手工维护汇总表、看板数据或公式字段。
- 不用工作流计算金额、利润、月份、数量。

### 阶段 3：展示看板层

目标是把已稳定的数据和联动结果展示成经营判断。

进入条件：

- 基础数据层已经确认表、字段、关联和历史数据处理方式。
- 录入与联动层已经确认用户录入字段、自动字段、状态字段和工作流边界。
- 至少有样例数据或可人工核对的历史数据。

必须输出：

- 看板问题：用户到底要看什么经营判断。
- 指标清单：销售额、订单数、利润、未收款、交付状态、费用结构等。
- 数据源：每个组件读哪张事实表或汇总表。
- 指标字段：每个组件聚合哪个字段，聚合方式是什么。
- 筛选字段：优先使用单选、显式状态、月份筛选、负责人等稳定字段。
- 组件类型：指标卡、趋势、排行、分布、异常列表。
- 验证方式：完整重载页面，并用至少两个筛选值切换验证。

看板设计规则详见 `dashboard-build-rules.md`。

### 阶段 4：验收发布层

目标是确认系统能被用户长期使用，而不是只建了结构。

必须验证：

- 用户日常视图只展示必要录入字段。
- 批量录入视图能按固定字段顺序从 Excel/旧表复制粘贴事实字段。
- 导入校验视图能发现关键异常。
- 自动字段不要求用户手填。
- 新增订单、收款、支出或交付记录后，相关公式、lookup、状态和工作流会联动。
- 看板组件完整重载后仍能渲染。
- 至少两个筛选值切换后，目标组件数值发生正确变化。
- 月度销售额、利润、订单数、未收款等关键口径能和样例数据对上。
- 旧系统没有被误改。

## 模板路径

用户只要模板时，不要切到独立模板技能；模板是构建过程的资源。模板也必须按四阶段输出：

- 基础数据层。
- 录入与联动层。
- 展示看板层。
- 验收发布层。

如果用户要参考飞书官方模板，先用 `official-template-design-patterns.md` 判断应该吸收哪类共性结构，再用 `official-template-reference-library.md` 查具体官方复制件或官网搜索入口。

## 是否先本地验算

飞书 Base 是默认落地载体。本地 Excel / 表格只能作为校验、预演和调整工具，不能替代飞书多维表格的长期系统能力。

直接进飞书 Base 的场景：

- 简单客户、订单、收款登记。
- 一人公司轻量经营系统。
- 模板已经成熟，字段和公式风险低。

先本地验算的场景：

- 财务结算、利润分成、库存核算复杂。
- 有大量历史数据迁移。
- 用户给出的口径存在冲突。
- 关键指标错了会影响对账、收款或经营决策。

本地 Excel / 表格只是验证工具，不是日常系统。它可以用于复杂口径验算、历史数据清洗预演、导入字段顺序设计和样例数据核对；验算完成后，最终入口仍应回到用户和 Agent 都能共用的飞书 Base 或类似 SaaS。

## 飞书实施规则

飞书 CLI 准备不单独作为技能暴露，它是构建流程的实施前置步骤。

只要用户目标是搭到飞书 Base，飞书 CLI 就是硬前置，不是可选项。

涉及飞书 Base 操作前，必须先确认 `lark-cli --version`、`lark-cli doctor` 和 `lark-cli auth status` 可用，并确认目标 Base 和当前身份具备修改权限。没有可用 CLI 时，只能输出方案、模板或本地验算结果，不能承诺已经搭建，也不能把本地 Excel 当成默认交付物。

涉及飞书 Base 操作时必须使用 `lark-base` skill，并按其要求读取对应 reference。创建公式字段前读 formula guide。创建 lookup 字段前读 lookup guide。不要猜字段 ID、表名、字段类型和公式语法。

## 输出格式

方案阶段按四段输出：

1. 基础数据层：表、字段、字段类型、关联关系、旧数据处理。
2. 录入与联动层：录入页、批量页、Agent 入口、校验视图、公式、lookup、状态、工作流。
3. 展示看板层：指标、数据源、指标字段、筛选字段、组件、验证方式。
4. 验收发布层：样例数据、验收动作、用户使用路径、残余风险。

搭建完成后输出：

1. 建了哪些表、视图、看板、工作流。
2. 用户日常怎么用。
3. 批量录入怎么用，字段顺序和校验视图在哪里。
4. 哪些字段不用填。
5. 样例数据和关键口径是否已核对。
6. 看板是否通过完整重载和双筛选值验证。
7. 后续明确待办。
