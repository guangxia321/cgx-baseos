# 业务数据诊断与 AI 数仓判断

## 何时读取

当用户说“想做数字化”“想接入 AI”“想搭经营系统”“不知道哪些数据有价值”“员工重复录入太多”“数据散在各处”时，先读这份 reference。

目标不是马上建表，而是判断：

- 企业里哪些数据最关键、最稳定、最值得沉淀。
- 哪些重复动作最占用员工时间，适合自动化或 Agent 代办。
- 哪些数据现在没有被分析，但会影响收款、利润、交付、复购和经营判断。
- 用户应该用人手填表、批量导入、Agent 代录，还是混合录入。
- 是否已经具备进入构建/重构的条件。

广厦 BaseOS 的核心不是替代 ERP、CRM 或 SaaS，而是帮助企业建立人和 AI 都能使用的汇总经营数仓。

## 第一层：请求类型

| 请求信号 | 归类 | 下一步 |
| --- | --- | --- |
| “想做数字化 / AI 接入 / AI 自动分析 / 企业 AI 基座” | 业务数据诊断 | 输出一页业务数据地图 |
| “不知道哪些数据有价值 / 哪些数据没利用上” | 数据资产盘点 | 识别关键对象、漏分析数据和 AI 可用入口 |
| “员工重复录入太多 / 流程太费时间” | 重复动作诊断 | 识别可简化录入、批量导入和自动化机会 |
| “从 0 搭一个销售/订单/客户/财务系统” | 构建意图，但先诊断 | 如果没有业务数据地图，先做诊断 |
| “旧表太乱，不好维护 / 有历史 Excel” | 重构意图 | 先盘点旧数据能否成为主数据或事实数据 |
| “图表不显示、公式错、筛选不动、工作流没触发” | 维修现有系统 | 切到 `repair-playbook.md` |
| “有没有模板 / 给我 Excel/飞书模板” | 构建资源请求 | 进入构建流程，模板作为资源读取 |
| “只是算一次数据/临时整理名单” | 不必搭系统 | 输出轻量表格或一次性处理方案 |

## 第二层：企业与业务类型

| 类型 | 典型特征 | 第一版优先诊断 |
| --- | --- | --- |
| 一人公司 | 创始人自己获客、销售、交付、收款，信息散在聊天、表格和脑子里 | 产品/服务、线索、客户、交付、收款、复购、渠道转化 |
| 企业服务团队 | 有线索、商机、方案、合同、项目、续费、客户成功 | 线索到回款、项目交付、客户成功、续费和项目利润 |
| 销售型小企业 | 有稳定客户、商品/服务、订单和收款 | 客户、商品/服务、订单、收款、费用、经营看板 |
| 项目制交付 | 每个客户有项目、任务、里程碑、交付物和回款节点 | 项目状态、交付节点、成本、回款和利润 |
| 轻库存/渠道业务 | 有商品、入库、出库、库存、渠道、门店 | 商品、库存流水、订单、渠道、盘点；第一版避免复杂 ERP |
| 混合经营 | 同时有销售、交付、财务、内容或渠道资产 | 只选最能减少人工和产生经营判断的一条主链路 |

## 第三层：数据价值判断

判断企业数据是否值得进入经营数仓，优先看四类价值：

| 价值 | 判断问题 | 示例 |
| --- | --- | --- |
| 稳定对象 | 哪些对象长期存在、会被反复引用 | 客户、产品/服务、项目、合同、渠道、财务科目 |
| 重复事实 | 哪些事情反复发生、需要留痕 | 订单、收款、支出、交付记录、跟进记录、库存变动 |
| 经营判断 | 哪些数字决定业务好坏 | 销售额、未收款、利润、交付进度、复购、渠道转化 |
| AI 入口 | 哪些数据适合让 Agent 查询、录入、提醒、总结 | 客户状态、项目风险、催收名单、月报、异常订单 |

用户主要维护主数据和事实数据。派生数据和展示数据应该通过公式、lookup、视图、看板或工作流自动联动。

## 第四层：是否进入构建

适合进入构建的信号：

- 已经有稳定对象，例如客户、商品/服务、订单、项目、合同、财务科目。
- 有重复动作，例如获客、报价、成交、交付、收款、复购、月报。
- 用户能说出最想长期看到的 3 个数字。
- 用户有持续录入意愿，或愿意让 Agent/批量导入承担录入。
- 后续希望 Agent 能录入、查询、汇总、提醒或生成报告。

暂时不适合完整构建的信号：

- 业务对象还不稳定，今天做 A，明天做 B。
- 只是一次性整理或一次性统计。
- 用户没有持续录入入口，也没有历史数据可沉淀。
- 关键口径没有共同含义，例如“收入”“成交”“完成”“利润”团队内不一致。

如果暂时不适合搭，输出最小替代方案：一张临时表、一个字段清单、一个 SOP 或一个手工验算表。

## 第一轮最小问题

不要一次问太多。优先问能决定数据地图的问题：

1. 你每天或每周最重复、最耗时的动作是什么？
2. 这些动作现在记录在哪里：飞书、Excel、SaaS、聊天记录、附件，还是人的脑子里？
3. 哪些字段最容易填错、漏填、格式不统一？
4. 你最想稳定看到的 3 个数字是什么？
5. 哪些数据一旦错了，会影响收款、利润、交付或客户关系？
6. 你希望谁来录入：人手填表、从旧表批量粘贴、Agent 代录，还是混合方式？
7. 现在有没有已经跑通、值得保留的旧表或模板？

## 输出要求

- 先讲清楚为什么这些数据值得沉淀，再谈是否构建。
- 不要把模板名当结论；模板只能作为后续构建资源。
- 不要用传统 ERP/SaaS 模块替代业务判断。
- 如果用户只需要一次性处理，明确不建议搭完整系统。
