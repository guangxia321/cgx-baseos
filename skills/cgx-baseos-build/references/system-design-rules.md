# 经营系统设计规则

## 何时读取

当用户要新建、重构或评审经营系统时读取这份 reference。它负责基础数据层、录入与联动层、字段分层和工作流边界。新建看板的详细规则读 `dashboard-build-rules.md`；维修已有看板先读 `repair-playbook.md`。

## 核心原则

用户录入成本第一。凡是能从客户库、商品库、订单事实、公式、关联、lookup、看板或工作流得到的数据，都不要让用户重复录入。

用户和 Agent 要共用同一套入口。系统不能依赖只有 Agent 才会执行的本地脚本、隐藏步骤或外部刷新动作作为日常前提。

四阶段顺序不能倒置：

1. 基础数据层：先定主数据、事实数据、历史数据、字段类型和关联关系。
2. 录入与联动层：再定录入页、批量页、Agent 入口、公式、lookup、显式状态和工作流。
3. 展示看板层：最后基于稳定字段设计看板，不倒推数据模型。
4. 验收发布层：用样例数据验证真实链路和用户使用方式。

## 数据分层

| 层级 | 说明 | 用户是否维护 |
| --- | --- | --- |
| 主数据 | 客户、商品/服务、品牌规则、财务科目、渠道账号等稳定对象 | 少量维护 |
| 事实数据 | 订单、收款、支出、项目、交付记录等真实发生的记录 | 主要维护 |
| 派生数据 | 公式字段、按订单号/月度聚合的汇总行、利润、未收款 | 自动生成 |
| 展示数据 | 看板、视图、报表、筛选器 | 自动读取 |

用户主要维护主数据和事实数据；派生数据和展示数据应自动联动。

## 基础数据层

销售类和经营类系统通常从这些表开始：

| 层级 | 表 | 说明 |
| --- | --- | --- |
| 主数据 | 客户管理 | 客户名称、地址、联系人、来源、状态 |
| 主数据 | 商品 / 服务管理 | 名称、品牌、规格、售价、利润率、状态 |
| 主数据 | 品牌规则 / 财务科目 | 利润率、科目、分类规则 |
| 事实 | 订单录入 | 唯一订单事实入口，一单多商品可多行同订单号 |
| 事实 | 收款记录 | 收款凭证、多渠道收款、部分收款 |
| 事实 | 财务流水 | 支出、费用、非订单收入 |
| 派生 | 订单汇总 | 按订单号聚合，不由用户手填 |
| 派生 | 月度经营汇总 | 按月份聚合，不由用户手填 |
| 展示 | 看板 | 经营、收款、客户、商品、财务 |

不同业务只需要替换主对象，例如课程、设备、服务、项目、渠道、门店等。默认先跑通轻量闭环，只有当销售周期、交付过程、库存、财务审计、售后 SLA 或目标管理真实存在时，才把结构加重。

## 录入与联动层

录入动作本身必须被重点设计。常见入口至少四类：

- 日常录入页：成员各自新增客户、订单、收款、任务或支出。
- 批量录入页：从 Excel/旧表复制粘贴，字段顺序固定。
- Agent 入口：Agent 代录、查询、提醒时使用同一套事实表和视图。
- 导入校验视图：检查缺客户、缺商品、金额为空、日期异常、状态未识别、重复编号。

主数据表也必须有录入入口。不要只建一张客户表或商品/服务表就结束；同一张主数据表通常需要多种使用形态：

- 登记 / 维护视图：给人新增和修改主数据，只展示名称、分类、价格、成本、状态、负责人、说明等维护字段。
- 可用目录视图：给订单、项目或 Agent 引用，只展示已启用、可销售、可交付的对象。
- 批量导入视图：当用户已有 Excel、历史清单或商品库时，字段顺序固定，用于批量粘贴。
- 表单入口：当录入人不是系统维护者，或需要手机端/外部人员提交时，在同一张表上创建表单，而不是新建一张“产品录入表”。

这类“子入口”是同一张表的不同视图或表单，不是默认新增子表。只有当录入内容已经变成独立事实动作，例如产品上架申请、调价记录、样品入库、服务变更审批，才单独建事实表。

订单录入必须足够少字段。典型用户输入字段：

- 订单号
- 订单日期
- 客户
- 商品 / 服务
- 数量
- 成交价或折扣
- 状态 / 备注 / 附件

典型自动字段：

- 自动月份
- 自动商品品牌 / 规格 / 售价
- 自动应收金额
- 自动利润
- 自动实收金额
- 自动未结清金额
- 结算状态
- 月份筛选

检查标准：用户新增订单时，只需要选客户、选商品或服务、填事实字段。凡是要求用户重复输入客户名称、商品名称、品牌、规格、利润率、地址的设计，都要先质疑。

## 字段与工作流边界

不要把快照字段当默认方案。客户名、商品名、品牌、售价、利润率、地址等，优先用关联、lookup 或公式读取。只有在业务明确要求历史记录不随主数据变化且有真实审计需求时，才考虑版本化或订单事实字段。

工作流只承担动作，公式承担计算。公式可以计算金额、利润、数量、月份、收款状态、月度指标；工作流适合补行、通知、反写、提醒。

适合工作流的事情：

- 新订单后补 `订单汇总` 行。
- 新订单或财务流水后补 `月度经营汇总` 行。
- 收款记录关联订单后反写订单收款字段。
- 未收款提醒、月报提醒、异常提醒。

不适合工作流的事情：

- 算金额、利润、数量、月份。
- 维护客户名、商品名、品牌名快照。
- 做看板常规刷新。

## 展示看板层边界

看板不应反向决定基础数据和录入设计。只有基础数据层和录入联动层稳定后，才进入展示看板层。

看板阶段必须明确：

- 指标回答什么经营问题。
- 组件读哪张事实表或汇总表。
- 指标字段和聚合方式是什么。
- 筛选字段是否为稳定单选、状态字段或月份筛选字段。
- 是否能完整重载并通过至少两个筛选值验证。

详细规则见 `dashboard-build-rules.md`。

## 沟通问题

当需求不清楚时，优先问这些问题：

1. 你最想少做哪一步人工操作？
2. 你每天/每周最常录入的是什么？
3. 你最终最想看的 3 个数字是什么？
4. 哪些信息一旦改错会影响历史数据？
5. 用户会自己在飞书里录，还是主要让 Agent 代录？
6. 有没有旧表已经跑通了某个功能？
7. 哪些模块第一版可以不做？

不要过早讨论权限、库存、复杂审批、组织架构，除非用户明确需要。

## 验收清单

搭完后必须检查：

- 用户日常视图是否只展示必要录入字段。
- 后台公式字段是否隐藏或明确标注不用填。
- 批量录入视图是否有固定字段顺序和校验视图。
- 客户、商品、订单、收款、财务的来源是否清楚。
- 订单新增后，订单汇总和月度汇总是否有对应行。
- 看板组件是否读取正确字段，没有读快照、缓存或外部临时表。
- 看板组件在页面完整重载后是否仍能正常渲染。
- 月度销售额、利润、数量、订单数是否能和样例数据或人工基线对上。
- 现有线上系统是否未被误改。

## 禁止事项

- 不要默认创建权限体系。
- 不要默认创建库存体系。
- 不要默认创建外部脚本作为日常必需步骤。
- 不要默认创建快照字段、看板缓存字段或看板数据源表。
- 不要让用户手工维护汇总表和看板数据。
- 不要在未确认前修改旧线上系统。
- 不要只建结构，不验证真实数据口径。
