# 飞书官方模板共性设计模式

## 何时读取

当用户要从 0 搭建系统、重构旧系统、参考官方模板、拆解行业模板，或希望把某个业务做成 BaseOS 经营底座时，读取这份 reference。

这份资料不是模板清单，而是从已复制并索引的 47 个飞书官方多维表格模板中提炼出的结构经验。具体模板入口看 `official-template-reference-library.md`；本文件用于决定“应该怎么建”，避免机械照搬某一个模板。

使用这份资料时，要同时看两件事：

- 相同点：官方模板反复证明哪些底层结构是稳定的。
- 差异点：不同业务为什么会多出某些表、状态、字段和看板。

最终目标是在遇到类似业务、职能板块或需求时，快速判断最小可用系统应该怎么搭，而不是记住某个行业模板的表名。

## 1. 官方模板共同底层结构

成熟官方模板虽然行业不同，但多数都落在同一套系统骨架里：

| 层级 | 常见表 | 作用 | 设计要点 |
| --- | --- | --- | --- |
| 主数据 | 客户、联系人、产品、服务、供应商、仓库、门店、人员、平台、达人、律师库 | 稳定对象库 | 少字段、高复用，作为其他表的关联源 |
| 业务事实 | 订单、合同、项目、任务、采购单、销售单、工单、案件、直播场次、上课登记 | 真实发生的业务记录 | 用户主要录入入口，保留时间、金额、数量、负责人、状态 |
| 跟进/流转 | 客户跟进、商机阶段、售后回访、投诉处理、项目日报、任务进展 | 记录过程变化 | 不覆盖历史，用一条条记录承载动作 |
| 财务/数量明细 | 收款、付款、发票、报销、预算、库存出入库、退换货 | 对账和核算依据 | 尽量和订单、合同、项目、库存对象关联 |
| 派生计算 | 订单汇总、月度统计、库存台账、项目回款汇总、目标完成率 | 自动计算结果 | 通过 lookup、rollup、formula 或工作流生成，不让用户手填 |
| 展示看板 | 销售看板、经营分析、库存看板、工厂看板、项目进度、工单满意度 | 经营判断入口 | 优先读取事实表和自动字段，避免单独维护看板数据 |
| 配置字典 | 分类、阶段、目标、支出类型、服务项目、门店列表、平台列表 | 控制选项和口径 | 只在确实需要跨表复用时独立成表 |

BaseOS 搭建时，先判断用户业务中的主对象和事实动作，再套用这套骨架。不要从“做多少张表”开始，而要从“谁在录入什么事实、最终要看什么结果”开始。

## 2. 通用业务对象地图

官方模板反复出现的对象可以抽象成一张 BaseOS 对象地图：

| 对象 | 常见来源模板 | 典型关系 | BaseOS 吸收方式 |
| --- | --- | --- | --- |
| 客户 | CRM、合同、订单、门店、课程销售、工单 | 关联联系人、商机、订单、合同、项目、工单 | 所有经营系统的默认主数据之一 |
| 联系人 | CRM、客户关系、供应商、律所协作 | 属于客户、供应商或合作方 | 不要把联系人塞进客户表的长文本里 |
| 线索/商机 | CRM、一站式 CRM、项目制 CRM | 线索转客户，商机关联客户、合同、销售负责人 | 适合销售周期较长的业务；轻量业务可先省略商机 |
| 产品/服务 | 进销存、门店、电商、制造、课程 | 关联订单、库存、价格、成本、服务次数 | 统一处理商品、服务、课程、项目包 |
| 合同/订单 | CRM、合同管理、销售订单、制造签单 | 关联客户、产品、收款、发票、交付项目 | 是收入链路核心事实表，优先标准化 |
| 项目/任务 | 项目管理、客户成功、制造项目、翻译项目 | 项目关联客户、合同、任务、预算、回款 | 适合交付周期长、多人协作或里程碑业务 |
| 收款/付款/发票 | 合同管理、财务明细、报销、进销存 | 关联订单、合同、采购单、项目 | 不要只在订单里写一个“已收款”文本 |
| 库存/仓库/供应商 | 进销存、电商样品、连锁门店、工厂仓储 | 产品、仓库、入库、出库、调拨、盘点 | 有真实库存压力时再进入第一版 |
| 工单/反馈/投诉 | 工单、售后、客户投诉、退换货 | 关联客户、产品、负责人、满意度 | 适合服务交付、售后、内部支持 |
| 人员/负责人 | 几乎所有模板 | 销售负责人、项目经理、客服、采购员、教练、律师 | 用 user 字段或人员表承载责任归属 |
| 目标/看板 | CRM 目标、销售目标、电商 GMV、工厂产能 | 关联时间周期、负责人、实际完成值 | 只在需要目标管理时单独建目标表 |

第一版系统通常不需要一次性覆盖所有对象。默认优先顺序是：客户/产品或服务/订单/收款/看板；有交付再加项目/任务；有实物再加库存/供应商；有售后再加工单/反馈。

## 3. 表类型设计模式

### 主数据表

主数据表保存稳定对象，不保存大量过程。典型字段包括：名称、分类、状态、负责人、联系方式、地址、备注、附件。

真实模板参考：

- `CRM与ERP进销存一体化管理系统`：客户管理、公司产品、供应商管理、仓库列表。
- `一站式CRM管理系统`：客户表、客户联系人。
- `进销存管理系统`：产品、客户、供应商、仓库。
- `电商行业一站式公司管理表`：样品信息、达人库、平台。
- `律所案件管理系统`：律师库。

### 业务事实表

事实表记录真实发生的一次动作或一笔业务。它应该是用户日常录入的主要入口。典型字段包括：编号、日期、关联对象、金额、数量、负责人、状态、附件。

真实模板参考：

- `一站式CRM管理系统`：合同订单、商机表、发票表。
- `进销存管理系统`：采购订单、销售订单、销售出库、采购入库。
- `项目管理全流程`：项目表、任务表、预算表、报销单。
- `门店管理系统`：门店预约、门店开单、划卡记录。
- `电商行业一站式公司管理表`：直播场次、销售明细、样品出入库。
- `制造业公司管理表`：签单日报、报价、合同、项目计划。
- `工单管理表`：工单总表、满意度收集。
- `课程销售客户关系管理系统`：成交登记、上课登记。
- `律所案件管理系统`：案件、任务。

### 跟进/流转记录表

当一个对象会反复沟通、推进、回访或处理，不要只在主表里覆盖“最新进展”。应使用单独记录表保留历史。

真实模板参考：

- CRM 模板普遍有 `客户跟进记录`。
- 项目模板有 `日报填写`、`周报`、`任务进展`。
- 售后模板有 `满意度回访`、`投诉回访`、`处理结果`。
- 门店模板有 `客户跟进` 和 `预约`。

BaseOS 默认规则：主表可保留“最近跟进时间”“当前状态”等自动字段，但完整过程放在流转记录表。

### 财务/数量明细表

只要涉及收款、付款、开票、报销、库存变化，就应有可对账的明细，而不是只写一个最终状态。

真实模板参考：

- `合同管理系统`：合同收支、合同发票、开票信息。
- `项目管理全流程`：预算表、报销单。
- `进销存管理系统`：采购入库、销售出库、采购退货、销售退货、付款单、收款单。
- `电商行业一站式公司管理表`：销售明细、样品出入库。
- `公司预算管理一体化系统`：预算控制、账单管理。

BaseOS 默认规则：金额状态可以公式计算，但收付款凭证、发票、库存出入库这类事实必须有明细记录。

### 派生统计表

派生表用于汇总，不应该成为用户手填入口。常见来源是订单、收款、库存、项目、工单。

真实模板参考：

- CRM 模板中的 `销售目标管理`、合同客户统计、发票开票统计。
- 项目模板中的 `项目汇总`、项目回款记录。
- 进销存模板中的 `实时库存`。
- 电商模板中的 `销售目标`、GMV/利润统计。
- 制造模板中的 `签单月报`、延期率统计。

BaseOS 默认规则：能从事实表算出来的字段，不让用户二次录入。只有当飞书原生能力无法补行时，才用工作流创建汇总行。

### 看板/目标表

看板是展示入口，不是数据来源。目标表只在需要按人、按月、按团队管理目标时创建。

真实模板参考：

- `一站式CRM管理系统`：销售目标管理。
- `销售收入统计仪表盘`：收入统计看板。
- `数字化智能工厂看板`：生产、仓储、设备指标。
- `课程销售客户关系管理系统`：销售看板。
- `工单管理表`：满意度和工单状态分析。

BaseOS 默认规则：先建事实字段和自动字段，再建看板；不要为了看板先造一张用户手填的统计表。

## 4. 字段类型设计模式

| 字段模式 | 推荐字段类型 | 适合承载 | 常见问题 |
| --- | --- | --- | --- |
| 编号 | 自动编号、公式 | 合同号、订单号、商机编码、工单号、客户编码 | 不要让用户手写唯一编号 |
| 名称 | 文本、公式标题 | 客户名、产品名、项目名、组合标题 | 不要把多个核心维度混在一个长文本 |
| 状态/阶段 | 单选、多选、复选框 | 销售阶段、付款状态、出库状态、任务状态、工单状态 | 不要用普通文本写状态，否则难筛选和统计 |
| 负责人 | 人员字段、人员表关联 | 销售、项目经理、客服、采购、教练、律师 | 不要只写姓名文本 |
| 时间节点 | 日期时间、创建时间 | 推荐日期、签约日期、预计结束、付款日期、上课日期 | 不要只保留月份文本 |
| 金额/数量 | 数字、货币、公式 | 合同金额、订单金额、库存数量、GMV、成本、利润 | 不要把金额写进备注 |
| 关联关系 | 关联记录 | 客户-联系人、客户-订单、订单-收款、产品-库存 | 不要重复录客户名、产品名 |
| 跨表读取 | 查找引用、汇总 | 读取客户负责人、产品价格、合同金额、库存汇总 | 查找字段作为筛选条件前要实际验证 |
| 自动计算 | 公式 | 应收、未收、完成率、延期天数、库存预警、利润 | 复杂公式要优先拆小字段 |
| 附件凭证 | 附件 | 合同扫描件、发票、报销凭证、产品图、名片 | 不要只贴外链且无结构字段 |
| 标签分类 | 单选、多选、字典表 | 来源、行业、服务分类、平台、紧急程度 | 选项要少而稳定，别把标签当备注 |
| 备注说明 | 多行文本 | 跟进内容、工单描述、案件信息、复盘 | 备注不能替代状态、金额、日期 |

字段设计顺序：先确定事实字段，再确定关联字段，然后补自动字段，最后才设计看板字段。

## 5. 不同行业的结构差异

| 场景 | 共同骨架 | 主要变体 | 参考模板 |
| --- | --- | --- | --- |
| CRM/销售 | 客户、联系人、线索/商机、合同/订单、跟进、收款、目标 | 商机阶段、客户来源、销售负责人、预测成交 | `CRM与ERP进销存一体化管理系统`、`一站式CRM管理系统`、`项目制公司CRM管理系统` |
| 销售订单合同 | 客户、合同、订单、发票、收款、收入统计 | 合同相对方、开票状态、付款状态、续费率 | `合同订单管理表`、`合同管理系统`、`销售订单及库存管理表` |
| 项目交付 | 项目、任务、周报、预算、报销、回款 | 进度、逾期、项目经理、交付成果、复盘 | `项目管理全流程`、`项目汇总经营分析管理表`、`客户成功团队项目管理表` |
| 财务经营 | 收入、支出、预算、报销、发票、月度统计 | 科目、凭证、审批状态、现金流、利润 | `经营明细数据分析`、`公司预算管理一体化系统`、`AI 发票识别与报销管理系统` |
| 进销存供应链 | 产品、客户、供应商、仓库、采购、销售、库存、收付款 | 入库、出库、退货、调拨、盘点、库存预警 | `进销存管理系统`、`采购仓储管理`、`库存出入库管理表` |
| 门店餐饮零售 | 门店、客户/会员、预约、开单、服务项目、库存、巡检 | 次卡、划卡、导流项目、门店巡检、连锁库存 | `门店管理系统`、`餐饮经营与管理全流程管理表`、`连锁门店进销存管理系统` |
| 电商直播内容 | 商品/SKU、平台、达人、直播场次、销售明细、样品库存、复盘 | GMV、退货、毛利、话术、内容排期、达人合作 | `电商行业一站式公司管理表`、`直播数据自动录入与复盘分析系统` |
| 制造生产 | 客户、报价、合同、项目计划、生产任务、仓储、人员 | 工序、设备、OEE、产能、延期率、质量 | `制造业公司管理表`、`数字化智能工厂看板` |
| 售后服务 | 客户、产品、工单、负责人、处理记录、满意度 | 紧急程度、期望完成、投诉回访、退换货 | `工单管理表`、`售后服务工单`、`客户投诉处理流程管理系统` |
| 教育/专业服务 | 客户、联系人、销售机会、合同/成交、服务交付、人员 | 学员、课程、上课登记、案件、律师、翻译文件 | `课程销售客户关系管理系统`、`律所案件管理系统`、`翻译公司项目管理表` |

行业差异通常体现在对象命名、状态阶段、数量口径和看板指标上。底层表型仍然是主数据、事实、过程、财务/数量、派生、展示。

## 6. 共通搭建思路与差异决策

### 共通搭建思路

看完这些模板后，最重要的共通搭建思路不是“行业分类”，而是下面这套顺序：

1. 先找主对象：谁是长期存在的对象，例如客户、商品、服务、项目、门店、供应商、人员。
2. 再找事实动作：用户每天真正新增的是什么，例如订单、合同、收款、任务、工单、库存流水、上课记录。
3. 再找流转过程：哪些东西会被反复跟进、推进、回访、复盘，例如客户跟进、项目日报、售后处理。
4. 再找核算闭环：这件事最后用什么对账，例如应收、实收、发票、成本、库存、工时、完成率。
5. 再找展示目标：老板、员工或 Agent 最终要看什么，例如未收款、项目进度、库存预警、销售目标、满意度。
6. 最后才决定自动化：公式负责计算，lookup 负责引用，工作流负责创建、提醒、反写，不把计算塞进工作流。

这套顺序比行业模板名更重要。CRM、门店、教培、律所、电商、制造，底层都是“主对象 + 事实动作 + 过程记录 + 核算闭环 + 看板输出”的组合。

### 核心差异点

不同模板真正的差异，不是界面长得不一样，而是业务闭环不同：

| 差异维度 | 轻量搭法 | 加重搭法 | 什么时候加重 |
| --- | --- | --- | --- |
| 销售周期 | 客户 + 订单 + 收款 | 线索 + 商机 + 跟进 + 合同 + 收款 | 客户从接触到成交需要多次推进 |
| 收入事实 | 订单一张事实表 | 合同、订单、发票、收款拆开 | 有合同归档、开票、部分回款或审计要求 |
| 交付过程 | 订单状态字段 | 项目 + 任务 + 周报 + 预算 + 复盘 | 交付周期长、多人协作、需要看进度 |
| 商品/服务 | 产品/服务主数据 | SKU、仓库、采购、出入库、盘点 | 有实物库存、成本核算或供应商管理 |
| 售后服务 | 订单备注或状态 | 工单 + 处理记录 + 回访 + 满意度 | 有持续售后、投诉、内部支持 SLA |
| 财务管理 | 收款/支出明细 | 预算、报销、发票、科目、月度汇总 | 要做利润、现金流、预算控制或凭证留存 |
| 目标管理 | 看板直接聚合事实表 | 目标表 + 完成率 + 负责人维度 | 要按人、团队、月份追目标 |
| 行业特殊性 | 用通用对象改名 | 增加行业专有对象 | 行业对象会改变录入动作或核算口径 |

判断规则：先用轻量搭法跑通核心闭环，只有当业务真的需要审计、协作、库存、目标或过程追踪时，才把结构加重。

### 相同点汇总

官方模板之间最稳定的相同点：

- 都把稳定对象和业务动作分开：客户、产品、人员这类对象不应该混在订单、工单、项目里手填。
- 都依赖关联记录建立系统，而不是靠文本复制：客户关联订单，订单关联收款，项目关联任务，产品关联库存。
- 都需要状态字段表达业务阶段：销售阶段、付款状态、出库状态、项目进度、工单状态、投诉处理状态。
- 都把金额、数量、日期结构化：合同金额、订单金额、库存数量、GMV、成本、付款日期、交付日期都应该可统计。
- 都用自动字段减少重复录入：自动编号、月份、完成率、延期天数、未收金额、库存预警。
- 都把看板建立在事实和自动字段上：销售额、回款、库存、满意度、进度、利润都来自底层记录。
- 都需要保留过程痕迹：客户跟进、任务推进、售后处理、直播复盘、案件进展不应该只覆盖最新状态。

### 差异点汇总

官方模板之间最关键的差异点：

- CRM 类模板重在“转化”：线索、商机、跟进、合同和销售目标是核心。
- 合同订单类模板重在“收入凭证”：合同、订单、开票、付款状态和客户归属是核心。
- 项目交付类模板重在“进度”：项目、任务、负责人、计划/实际时间、预算和逾期是核心。
- 财务类模板重在“口径”：收入、支出、预算、报销、发票、科目和月度统计是核心。
- 进销存类模板重在“数量闭环”：采购、入库、销售、出库、退货、调拨、盘点和实时库存是核心。
- 门店类模板重在“到店动作”：客户/会员、预约、开单、服务项目、划卡、门店和导流是核心。
- 电商直播类模板重在“场次和商品”：直播场次、商品/SKU、达人、平台、样品库存、GMV、退货和复盘是核心。
- 制造类模板重在“生产履约”：报价、合同、生产计划、工序、设备、产能、质量和延期率是核心。
- 售后工单类模板重在“问题闭环”：工单、负责人、紧急程度、处理记录、回访和满意度是核心。
- 教育/专业服务类模板重在“人和服务记录”：学员/客户、课程/案件/项目、服务人员、成交、上课或交付记录是核心。

### 遇到类似需求时的快速判断

不要先问“有没有某行业模板”，先问这 5 个结构问题：

1. 这个业务最小收入闭环是什么：线索成交、订单收款、项目回款、门店开单，还是直播销售。
2. 用户每天新增的第一条记录是什么：客户、订单、任务、工单、库存流水、上课记录，还是销售明细。
3. 是否需要过程追踪：如果没有，就不要先建复杂流转表；如果有，就把过程从主表拆出来。
4. 是否需要核算闭环：涉及收款、发票、预算、库存、工时，就要有明细表。
5. 看板到底服务谁：老板看经营，销售看跟进，交付看进度，财务看对账，运营看转化。

回答完这 5 个问题，再从官方模板里找相似案例，通常就能判断最优搭法。

## 7. BaseOS 搭建时的默认生成规则

### 需求映射

听到用户需求后，先映射到对象和事实动作：

| 用户说法 | 默认映射 | 第一版优先表 |
| --- | --- | --- |
| 管客户、跟销售、提高转化 | CRM 链路 | 客户、联系人、线索/商机、跟进、合同/订单 |
| 管订单、回款、合同、发票 | 收入链路 | 客户、合同/订单、收款、发票、收入看板 |
| 管项目、交付、任务、进度 | 交付链路 | 项目、任务、日报/周报、预算/成本、项目看板 |
| 管支出、报销、利润、预算 | 财务链路 | 财务流水、预算、报销、发票、月度汇总 |
| 管商品、采购、库存、供应商 | 供应链链路 | 产品、供应商、仓库、采购、销售、库存流水 |
| 管门店、会员、预约、开单 | 门店链路 | 客户/会员、预约、开单、服务项目、门店看板 |
| 管直播、电商、达人、复盘 | 电商链路 | 商品/SKU、平台、达人、直播场次、销售明细、复盘 |
| 管售后、投诉、内部支持 | 服务链路 | 工单、客户、产品、处理记录、满意度 |

### 建表顺序

1. 先建主数据：客户、产品/服务、人员、供应商、仓库等。
2. 再建事实入口：订单、合同、项目、任务、工单、收款、库存流水。
3. 补过程记录：跟进、回访、日报、投诉处理、复盘。
4. 补自动字段：编号、月份、状态计算、金额计算、进度计算、未收金额。
5. 补汇总和看板：目标、月度统计、库存台账、经营看板。
6. 最后才考虑工作流：自动补行、通知提醒、反写状态。

### 第一版裁剪

第一版默认不把官方模板完整搬过来。按下面规则裁剪：

- 没有多人销售过程时，不建复杂商机阶段。
- 没有真实库存压力时，不建完整进销存。
- 没有合同审批流程时，不建复杂审批字段。
- 没有财务对账需要时，不建多层预算和科目。
- 没有稳定目标管理时，不建销售目标表。
- 没有持续售后动作时，不建完整工单闭环。

### 输出结构

给用户输出结构时，必须区分：

- 用户每天填什么。
- 系统自动算什么。
- 看板从哪里读。
- 哪些官方模板可作为参考。
- 哪些官方模板结构不建议直接照抄。

## 8. 官方模板反模式与避坑

官方模板可以参考，但不要无脑复刻：

- 不要照搬行业过拟合字段。餐饮、律所、制造、电商里的专有字段，只有用户业务真的需要时才引入。
- 不要一上来复制全量 ERP。`CRM + ERP + 进销存` 类模板结构完整，但第一版经常太重。
- 不要把看板当数据源。看板应读取事实表、自动字段或派生表。
- 不要让用户维护汇总表。订单汇总、月度汇总、库存台账应自动生成或自动计算。
- 不要用普通文本表达关键状态。付款状态、出库状态、销售阶段、工单状态应使用单选、复选框或公式。
- 不要把金额、数量、日期写在备注里。可统计字段必须结构化。
- 不要滥用 lookup 和公式。复杂公式优先拆成多个中间字段，并在样例数据上验算。
- 不要为了 AI 演示复制噪音表。AI 分析、AI 发票识别、自动复盘要先拆数据结构，再决定是否需要自动化。
- 不要默认匿名互联网访问一定可用。飞书公开权限可能仍受租户策略或账号态影响。
- 不要只看官方描述。真正有价值的是表名、字段、关联、视图、看板、工作流。

## 9. 如何把用户需求映射到参考模板

使用流程：

1. 先用本文件判断用户属于哪条业务链路，以及第一版该保留哪些对象。
2. 再到 `official-template-reference-library.md` 查询具体官方模板案例。
3. 优先选 2-3 个结构接近的模板对照，而不是只选一个模板照搬。
4. 从模板里吸收表型、字段模式、状态口径和看板指标。
5. 用 BaseOS 的四层数据模型重新组织：主数据、事实数据、派生数据、展示数据。
6. 如果现有 47 个模板没有合适案例，再去飞书官方模板库搜索关键词。

示例：

- 用户要“销售 + 合同 + 回款”：参考 CRM、一站式 CRM、合同管理系统，吸收客户、联系人、商机、合同订单、收款、发票结构。
- 用户要“项目制公司经营”：参考项目管理全流程、项目汇总经营分析、项目制 CRM，吸收项目、任务、合同、预算、回款、复盘结构。
- 用户要“门店预约和开单”：参考门店管理系统、餐饮经营全流程、连锁门店进销存，吸收会员、预约、服务项目、开单、划卡、门店看板结构。
- 用户要“电商直播复盘”：参考电商一站式管理、直播数据复盘、电商进销存，吸收商品/SKU、达人、平台、直播场次、销售明细、样品库存结构。
- 用户要“售后工单”：参考工单管理、售后服务工单、客户投诉处理，吸收工单、客户、产品、处理记录、满意度和回访结构。

最终目标不是让技能记住 47 个模板名称，而是让技能在面对新业务时能自动识别：哪些对象是主数据，哪些动作是事实，哪些字段应该自动算，哪些看板应该从哪里读。
