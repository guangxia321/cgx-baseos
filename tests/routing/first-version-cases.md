# 第一版路由与诊断测试用例

这些用例用于检查主入口是否能正确判断业务场景。测试重点不是回答得漂亮，而是不能走错路径。

## 用例 1：想做 AI 数字化

用户输入：

> 我想让公司业务接入 AI，但不知道哪些数据有价值。

期望：

- 归类为业务数据诊断。
- 路由到 `cgx-baseos-diagnose`。
- 读取 `business-classifier.md` 和 `high-frequency-scenarios.md`。
- 输出一页业务数据地图。
- 不能直接进入建表或模板输出。

## 用例 2：重复流程太重

用户输入：

> 我们员工每天都在整理 Excel、核对客户和收款，太花时间了。

期望：

- 归类为重复动作诊断 + 自动化机会识别。
- 路由到 `cgx-baseos-diagnose`。
- 识别重复动作、已有数据位置、AI 可用数据和推荐录入方式。
- 输出第一版最小构建范围。

## 用例 3：一人公司诊断产物

用户输入：

> 我是一人公司，客户、订单、交付和收款都在微信和几个表里，想整理一下。

期望：

- 路由到 `cgx-baseos-diagnose`。
- 诊断产物必须包含关键业务对象、重复动作、已有数据、AI 价值、录入方式和第一版构建范围。
- 不直接输出复杂 ERP 或团队权限方案。

## 用例 4：已有业务数据地图后构建

用户输入：

> 我已经有一页业务数据地图了，第一版主链路是客户、订单、收款和经营看板，帮我搭飞书 Base。

期望：

- 归类为构建/飞书落地。
- 路由到 `cgx-baseos-build`。
- 读取 `build-playbook.md`、`system-design-rules.md`、`dashboard-build-rules.md`、`feishu-cli-setup.md`。
- 输出必须按基础数据层、录入与联动层、展示看板层、验收发布层组织。
- 先检查 CLI、授权、目标 Base 和权限。
- 不能路由到 `cgx-baseos-feishu`。

## 用例 5：只要模板

用户输入：

> 你先给我一个一人公司的 Excel 模板。

期望：

- 归类为构建资源请求。
- 路由到 `cgx-baseos-build`。
- 可读取 `skills/cgx-baseos-build/templates/solo-business/schema.md`。
- 输出适用边界，并按四阶段说明基础数据、录入联动、展示看板和验收发布。
- 不能路由到 `cgx-baseos-templates`。

## 用例 6：经营财务

用户输入：

> 我想知道每个月收入、支出、利润，还有哪些客户没付款。

期望：

- 归类为构建中的经营财务场景。
- 路由到 `cgx-baseos-build`。
- 读取 `finance-scenarios.md`。
- 可读取 `skills/cgx-baseos-build/templates/finance-ops/schema.md`。
- 明确不替代专业会计、报税、总账。
- 不能路由到 `cgx-baseos-finance`。

## 用例 7：旧系统重构

用户输入：

> 我现在有一个多维表格，但是字段太多太乱，想重新整理。

期望：

- 归类为重构系统。
- 如果没有一页业务数据地图，先补诊断；如果已有地图，路由到 `cgx-baseos-build`。
- 先只读扫描旧表、字段、公式、视图、工作流和看板。
- 标记已跑通结构，优先保留。
- 不直接覆盖旧 Base。

## 用例 8：企业服务交付

用户输入：

> 我们是代运营团队，想管线索、报价、合同、项目进度和续费。

期望：

- 如果用户还没明确关键数据和录入方式，先路由到 `cgx-baseos-diagnose`。
- 一页业务数据地图应识别线索 -> 商机 -> 合同 -> 项目交付 -> 收款 -> 续费。
- 构建阶段可读取 `skills/cgx-baseos-build/templates/service-business/schema.md`。
- 不只做销售 CRM，也不能忽略交付和回款。

## 用例 9：项目利润

用户输入：

> 我想知道每个客户项目到底赚了多少钱，外包和广告费都要算进去。

期望：

- 归类为构建中的项目交付 + 经营财务。
- 路由到 `cgx-baseos-build`。
- 读取 `finance-scenarios.md`。
- 如果分摊规则复杂，先本地验算。
- 输出项目、收款、成本流水、项目利润的结构。

## 用例 10：看板筛选不动

用户输入：

> 飞书看板月份筛选点了以后，销售额和排行都不变。

期望：

- 归类为维修现有系统。
- 路由到 `cgx-baseos-repair`。
- 读取 `repair-playbook.md`、`dashboard-filter-slicer-case.md`。
- 先查数据源、筛选字段、组件联动规则。
- 不重建整套系统，不默认新增缓存表。

## 用例 10A：新建经营看板

用户输入：

> 我要搭一个销售系统，并且看本月销售额、利润、未收款和客户排行。

期望：

- 归类为构建中的展示看板层。
- 路由到 `cgx-baseos-build`。
- 读取 `dashboard-build-rules.md`。
- 不能路由到独立看板技能。
- 不能在基础数据层和录入与联动层未确认前直接输出看板组件。
- 看板阶段必须声明数据源、指标字段、筛选字段、组件类型和验证方式。

## 用例 11：图表配置变更

用户输入：

> 图表报“配置数据发生变更，请重新配置”。

期望：

- 归类为看板组件维修。
- 路由到 `cgx-baseos-repair`。
- 读取 `dashboard-troubleshooting.md`。
- 先查 block 配置、字段、筛选、旧配置残留。
- 必须页面完整重载验证。

## 用例 12：字段状态混乱

用户输入：

> 我们现在用未收金额大于 0 来筛未结清客户，但看板经常不准。

期望：

- 归类为字段建模 + 看板维修。
- 路由到 `cgx-baseos-repair`。
- 读取 `field-modeling-antipatterns.md` 和 `repair-playbook.md`。
- 建议补显式结算状态字段。
- 不继续堆复杂金额筛选。

## 用例 13：暂不适合完整建系统

用户输入：

> 我还没确定业务，可能卖课，也可能做咨询，也可能做工具，你先给我搭个全套系统。

期望：

- 判断暂不适合搭完整系统。
- 路由到 `cgx-baseos-diagnose`。
- 先要求明确稳定对象、高频动作和关键指标。
- 可输出轻量字段清单或临时表，不直接建复杂系统。
