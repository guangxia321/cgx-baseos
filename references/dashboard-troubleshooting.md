# 看板配置兼容排障

## 何时读取

遇到下面这些情况时，先读这份 reference：

- 用户反馈图表报 `配置数据发生变更，请重新配置`
- `dashboard-block-update` / `dashboard-block-create` 返回成功，但页面仍报错或空白
- 用户反馈月份、状态、客户等筛选条件选了以后，指标卡和排行都不变化
- 同一张表里的其他图表正常，只有某个组件持续失败
- 需要判断问题是字段坏了、配置坏了，还是飞书图表兼容性问题

如果问题重点是“筛选器不联动”“月份筛选不动”“跨表组件筛选不生效”，先读同目录下的 [dashboard-filter-slicer-case.md](dashboard-filter-slicer-case.md)。

## 常见原因

### 1. 结构变更

- 表被换了
- 字段被删了、改名了、改类型了
- 组件原来依赖的视图、字段或分组方式已经不成立

这类问题先查真实表结构，不要先怪缓存。

### 2. 组件配置漂移

服务端接受了配置，但飞书前端图表编辑器无法重新解释这份配置。常见触发点：

- 筛选器绑定的表和图表组件的数据源表不一致
- 组件切换过数据源，但筛选器的联动规则还残留旧表字段
- 用复杂公式字段做数字筛选
- 用 lookup / 关联展开字段做筛选或排序
- 旧配置里残留了当前前端不认的排序、筛选或分组结构
- 业务上等价，但图表序列化方式不兼容

### 3. 更新残留

`dashboard-block-update` 不是整块重写，而是 `data_config` 顶层 merge。  
这意味着：

- 你不传 `filter`，旧 `filter` 可能还在
- 你只改一部分配置，另一个坏配置可能继续残留

遇到这种情况，不要以为“这次命令里没写，就是没了”。

### 4. 前端状态滞后

后台已经改好，但当前页面还停在旧状态。  
所以“我刷新了一下”不一定足够，必须同时确认：

- 服务端当前 block 配置
- 页面完整重载后的实际渲染结果

### 5. 状态建模过弱

一些业务把“未收款”“部分收款”“仍有待收”只编码在金额比较里，比如：

- `<自动未结清金额字段> > 0`
- `<自动剩余数量字段> > 0`

这种建模在公式上可算，但在图表筛选层不一定稳定。  
更稳的做法通常是显式状态字段，比如：

- `<结算状态字段>`
- `<履约状态字段>`
- `<是否仍未结清字段>`
- `<时间标签字段>`

## 排障 SOP

### 第一步：先读服务端现状

必须先查：

- `dashboard-block-list`
- `dashboard-block-get`
- `field-list`
- `field-get`

先确认真实存在的组件、字段、类型、筛选、排序，再决定下一步。  
不要直接根据网页截图猜。

如果用户说“筛选不动”，还要额外列出每个组件的 `table_name`。同一个筛选器想控制多个组件时，先确认这些组件是不是同一张数据源表。

### 第二步：拆变量，不要一把梭重配

优先把问题拆成这四类：

1. `series` 坏了
2. `group_by` 坏了
3. `filter` 坏了
4. 旧配置残留坏了

方法是创建最小验证组件：

- 同一张表
- 同一个分组字段
- 同一个指标字段
- 先不加筛选

如果最小组件能渲染，说明字段本身未必有问题，继续一层层加回配置。

### 第三步：一次只改一个变量

推荐顺序：

1. 无筛选
2. 加业务状态筛选
3. 再加排序
4. 最后才加复杂数值筛选

不要同时改表、字段、排序、筛选、图表类型，不然排不出根因。

### 第四步：区分“能聚合”和“能筛选”

重点判断：

- 字段作为图表数值来源是否正常
- 字段作为图表筛选条件是否正常

很多问题恰好出在这里：  
字段能做 `series`，但一旦拿去做 `filter` 就触发兼容 bug。

还要区分“字段显示名相同”和“字段来源相同”。不同表里都叫 `月份` 的字段，不一定能被同一个筛选器联动。

### 第五步：双重验证

每次修改后都要做两类验证：

1. API 验证：确认服务端 block 结构已变更
2. 页面验证：完整重载页面，看组件是否真实渲染

只有两边都通过，才能算修好。

### 第六步：清理临时产物

排查过程中如果加了：

- 临时组件
- 临时字段
- 临时视图

结论确认后要删掉。  
否则后面会污染系统结构，用户也分不清哪个是正式组件。

## 稳定解决方案优先级

### 优先级 1：保留公式指标，改用稳定状态字段做筛选

这是最优先的方案。  
典型做法：

- `<自动金额指标字段>` 继续做图表数值
- `<显式状态字段> = 某状态` 改成图表筛选

这样通常既保留原业务口径，又避开飞书对复杂数字筛选的兼容问题。

### 优先级 2：如果业务有部分状态，先补显式状态字段

如果业务存在：

- 部分收款
- 部分发货
- 部分完成

就不要只靠“大于 0”这类数值比较做筛选。  
应该优先补一个更清楚的状态字段，例如：

- `结算状态 = 未结清 / 部分结清 / 已结清`
- `是否仍未结清 = 是 / 否`

如果旧模型没有这个字段，先补模型，再补图表。

### 优先级 3：改不干净就重建组件，不要反复 patch

如果怀疑旧配置残留，或者多次 patch 后行为不一致：

- 直接删掉这个 block
- 用已验证过的最小稳定配置重新创建

有时重建一个组件，比继续 patch 一个脏组件更稳。

### 优先级 4：最后才考虑新增中间表或缓存表

只有在下面条件同时成立时，才考虑额外数据源表：

- 飞书原生图表确实无法表达该查询
- 已经尝试过稳定状态字段、公式字段、汇总表字段
- 用户接受后续维护成本

默认不要因为一个图表报错，就把系统改成“多一张缓存表 + 多一步同步”。

## 这类问题的抽象结论

对别的业务、别的团队，这类问题通常不是“飞书不能做看板”，而是下面三件事之一：

1. 数据模型把业务状态藏在复杂数值里，没有显式状态字段
2. 图表组件吃到了前端兼容性差的字段组合
3. 排障时没有把“字段问题”和“筛选问题”拆开验证

所以长期解法不是“会修这个报错”，而是：

- 指标字段和筛选字段分层设计
- 关键业务状态显式建模
- 看板上线前做一次完整重载验证

## 可复用经验

一个高频规则可以直接抽象出来：

- 某个自动金额字段作为图表指标值可能是正常的
- 同一个字段一旦被拿去做数字筛选，可能触发组件报错
- 改成显式状态字段筛选后，图表通常会恢复正常
- 某个筛选字段在 A 表可用，不代表它能控制读取 B 表的组件
- 月份、状态这类用户常点选的维度，优先建成单选或明确状态字段，并在录入链路中自动维护

因此，以后遇到类似报错时，先不要怀疑整张表，也不要立刻加缓存表。  
先把筛选从“复杂金额比较”迁到“显式状态字段”，通常是最快、最稳的修法。
