# 高频场景库

## 何时读取

业务判断后读取这份 reference，用来确定第一版该搭什么，不该搭什么。高频场景不是技能拆分清单，而是主入口和搭建流程的路由依据。

## 第一批高频场景

| 场景 | 用户常见说法 | 第一版核心对象 | 关键结果 |
| --- | --- | --- | --- |
| 销售 / 客户 | “客户信息太乱”“我要登记客户和跟进” | 线索、客户、联系人、跟进、商机 | 客户状态、跟进进度、成交转化 |
| 订单 / 商品 / 服务 | “我要登记订单”“商品价格和利润要自动算” | 商品/服务、价格表、订单明细、订单汇总 | 销售额、订单数、商品排行、订单利润 |
| 批量录入 / 旧表导入 | “我平时从 Excel 复制过去”“旧表数据要快速录进去”“多人填得格式不统一” | 标准事实表、批量录入视图、导入校验视图、主数据匹配 | 快速粘贴、字段标准化、异常校验、自动联动 |
| 收款 / 应收 | “谁还没付款”“欠多少钱看不清” | 收款记录、订单、客户、凭证 | 已收、未收、逾期、催收名单 |
| 经营财务 | “钱花到哪里了”“本月利润多少” | 财务流水、财务科目、收款、支出、月度汇总 | 收入、支出、利润、现金流、费用结构 |
| 项目交付 | “成交后交付没人管”“项目进度看不清” | 项目、任务、里程碑、交付物、客户反馈 | 交付状态、逾期任务、项目利润 |
| 经营看板 | “我想看销售额/利润/未收款/排行” | 事实表、自动字段、筛选维度、看板组件 | 指标卡、趋势、排行、异常提醒 |
| 旧系统重构 | “表太乱了”“字段很多但不好用” | 现有表、字段、公式、视图、工作流、看板 | 复用可用结构，清理重复和快照 |
| 系统维修 | “图表不显示”“筛选不动”“公式错了” | 出问题的字段、组件、工作流或记录 | 最小修复、复现验证、保留线上系统 |

## 场景选择原则

第一版只抓一条最有价值的主链路。优先级一般是：

1. 最频繁发生的动作。
2. 最容易出错或漏掉的动作。
3. 最影响收款、利润、交付或经营判断的动作。
4. 最适合由用户和 Agent 共用入口录入的动作。
5. 最适合通过 Excel/旧表批量粘贴标准化的动作。

不要因为用户提到某个名词就扩大范围。例如用户说“想做 ERP”，但真实问题是“订单和收款看不清”，第一版应优先做订单、收款和经营看板，而不是权限、审批、库存、采购、人事全套系统。

## 场景到模板的映射

| 场景 | 优先模板 |
| --- | --- |
| 销售 / 客户 + 订单 / 收款 | `templates/sales-system/schema.md` |
| 一人公司销售、交付、收款 | `templates/solo-business/schema.md` |
| 企业服务线索到交付 | `templates/service-business/schema.md` |
| 收款、支出、利润、月报 | `templates/finance-ops/schema.md` |

模板是资源，不是独立技能。只有当用户明确要模板，或搭建流程已经判定模板适用时，才读取对应模板。

## 场景边界

- 销售系统不等于 CRM 全家桶。第一版先保证客户、跟进、订单、收款闭环。
- 财务系统不等于会计系统。第一版先解决经营可见性，不替代专业记账、报税、总账。
- 项目交付不等于完整项目管理软件。第一版先保证项目状态、任务、里程碑和回款节点清楚。
- 经营看板不是额外手工汇总表。优先读事实表和自动字段。
- 维修不是重构。只有局部修复解释不了问题，才扩大到模型调整。
