# 广厦 BaseOS 主入口

本技能只负责路由和边界判断。开发原则是少就是多：不要把经营财务、模板、看板新建、飞书 CLI 准备暴露成独立技能，它们都是诊断和构建过程中的内部能力。

定位上不要按 ERP、CRM、OA、BI 等传统模块切分需求，而是按业务主链路、经营数据资产和 AI 可调用性判断下一步。

## 三段式路由

| 用户请求 | 使用技能 | 说明 |
| --- | --- | --- |
| 想做数字化、AI 接入、AI 数仓、企业数据资产、流程自动化，但还不知道从哪里开始 | `$cgx-baseos-diagnose` | 先诊断关键数据、重复动作、AI 可用数据和自动化机会 |
| 想知道哪些数据没被利用、哪些流程占用员工时间、哪些指标最关键 | `$cgx-baseos-diagnose` | 输出一页业务数据地图，不急着建表 |
| 已有一页业务数据地图，或已经确认要搭销售、订单、客户、交付、收款、经营财务、看板 | `$cgx-baseos-build` | 基于地图按四阶段构建或重构最小可用经营系统 |
| 有旧表、历史 Excel、现有 Base，需要整理成新的经营数据底座 | `$cgx-baseos-build` | 先只读盘点旧数据，再决定复用、清洗和迁移路径 |
| 只要字段清单、Excel/飞书模板、经营财务结构、看板新建、或方案已确认要落到飞书 Base | `$cgx-baseos-build` | 模板、财务、看板新建和飞书 CLI 准备都属于构建内部步骤 |
| 图表不显示、筛选不动、公式错、字段写不进去、工作流没触发 | `$cgx-baseos-repair` | 维修已有系统，先局部定位，不默认重建 |

## 边界

- 第一步默认不是新建或重构，而是判断企业最关键的数据、重复动作、AI 可用数据和录入方式。
- 核心目标不是替代 ERP 或 SaaS，而是帮助企业沉淀能被人和 Agent 共用的经营数仓。
- 传统 ERP/CRM/OA/BI 分类只作为理解背景，不作为第一版系统边界。
- 模板是资源，不是独立技能；经营财务和看板新建是构建场景，不是独立入口；飞书 CLI 是实施前置步骤，不是独立技能。
- 维修类问题保持独立，不默认回到诊断或重构；只有根因指向底层模型错误时，才建议扩大范围。

## 输出

路由时说明：当前归类、应使用的子技能、下一步要读取的 reference/template、是否已有一页业务数据地图、少数待确认问题。
