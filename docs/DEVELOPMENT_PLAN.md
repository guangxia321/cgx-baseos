# 广厦 BaseOS 开发方案

## 当前目标

广厦 BaseOS 要做成一个低门槛的经营数字化技能。用户不需要先懂数据库、ERP、SOP 或 AI 基座，只需要说清楚自己的业务，技能负责先诊断关键数据和重复动作，再决定是否构建、如何构建、是否落到飞书 Base。

第一版外显结构固定为：

- `cgx-baseos`：主入口。
- `cgx-baseos-diagnose`：业务数据诊断。
- `cgx-baseos-build`：按四阶段构建、重构、模板、经营财务、看板新建、飞书实施。
- `cgx-baseos-repair`：维修排障。

## 产品思路

核心不是“帮用户建几张表”，也不是替代某个 SaaS，而是帮用户把经营动作沉淀成 AI 可以接入的数据底座。

技能要先搞清楚七件事：

1. 用户的业务形态是什么。
2. 最关键、最稳定、最值钱的数据是什么。
3. 高频重复动作是什么。
4. 哪些数据现在没有被分析。
5. 哪些流程最占用员工时间，适合简化录入或自动化。
6. 谁会录入：人、批量导入、Agent，还是混合方式。
7. 第一版最小主链路是什么。

## 主流程

### 1. 主入口路由

| 用户场景 | 处理方向 |
| --- | --- |
| 想做数字化、AI 接入、AI 数仓、企业数据资产 | 路由到 `cgx-baseos-diagnose` |
| 想知道哪些数据有价值、哪些流程可自动化 | 路由到 `cgx-baseos-diagnose` |
| 已有一页业务数据地图，要搭系统或重构旧表 | 路由到 `cgx-baseos-build` |
| 只要模板、经营财务结构、看板新建、或方案确认要落飞书 | 路由到 `cgx-baseos-build` |
| 看板、字段、公式、筛选器、工作流出错 | 路由到 `cgx-baseos-repair` |

### 2. 业务数据诊断

诊断阶段固定输出一页业务数据地图，包含：

- 业务类型与当前阶段。
- 关键业务对象。
- 重复动作与自动化机会。
- 已有数据与可迁移资产。
- 漏分析数据与关键指标。
- 推荐录入方式。
- 第一版最小构建范围。
- 下一步是否进入构建。

诊断阶段不要直接建表。用户业务还不稳定时，输出轻量表格、字段清单或 SOP。

### 3. 构建与重构

构建技能承接业务数据地图，处理两种主要场景：

- 有旧表或已有数据：先只读扫描旧表、字段、公式、lookup、视图、工作流、看板和历史记录，再判断哪些能复用、清洗或迁移。
- 没有数字化基础：以地图里的关键对象和重复动作为底板，先搭第一版最小主链路。

构建技能内部按四阶段推进：

1. 基础数据层：确认主数据、事实数据、历史数据、字段类型和关联关系。
2. 录入与联动层：设计日常录入页、批量录入页、Agent 入口、校验视图、lookup、公式、显式状态和必要工作流。
3. 展示看板层：只在前两层稳定后设计看板，明确指标、数据源、筛选字段、组件和重载验证。
4. 验收发布层：用样例数据验证录入、联动、看板、权限、旧系统未误改和用户日常使用路径。

模板、经营财务、看板新建和飞书 CLI 准备都是构建内部能力，不再切到独立技能。

### 4. 维修

维修路径独立于诊断和构建。先定位具体表、字段、公式、lookup、视图、看板组件、筛选器、工作流或记录。先复现，再最小修复，最后验证。

只有局部排障解释不了问题时，才建议回到诊断或构建层。

## 近期开发顺序

### M1：结构收敛

- 已把公开技能收敛为主入口 + 诊断 + 构建 + 维修。
- 已停止公开打包经营财务、模板输出和飞书实施技能。
- 已让构建技能吸收这些内部能力。
- 已把构建技能内部流程阶段化为基础数据、录入联动、展示看板、验收发布。

### M2：诊断样例

- 补 3-5 个真实或模拟的一页业务数据地图。
- 校准一人公司、企业服务、销售型小企业三类诊断问题。

### M3：构建验收

- 用业务数据地图驱动销售、交付、收款和经营财务模板。
- 补充旧表重构、历史数据迁移、批量录入校验案例。
- 补充看板新建的指标、数据源、筛选字段和双筛选值验证案例。

### M4：维修能力

- 持续沉淀看板、字段、公式、lookup、工作流问题案例。
- 增加典型失败案例和回归测试。
