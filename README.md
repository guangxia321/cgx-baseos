# 广厦 BaseOS（CGX BaseOS）

面向企业服务商和小企业老板的 AI 经营数仓搭建 Skill。

它帮你用飞书多维表格或类似低代码数据工具，先搭出一套最小经营系统：客户、线索、订单、项目、收款、费用、交付、看板和 AI Agent 可调用的数据入口。

可在 Codex、Claude Code、Cursor、Trae Solo 等支持 Skill / system prompt 的 Agent 上使用。

**最新版本：v1.0.0**

**作者**：[X](https://x.com/chenguangxia) · [抖音](https://v.douyin.com/bzhEngcV20k/)

> 很多企业的第一次真正数字化，是因为接触了 AI。  
> 然后发现 AI 用不上。  
> 因为数字化还没做好。

**作者背景**：笔者既做实体门店和经销业务，也做企业服务。在服务不同行业客户时，我发现很多企业并不是不会用 AI，而是还没有完成最基础的数据化和数字化。关键数据散落在微信、Excel、旧 SaaS、员工个人表格和口头同步里，AI Agent 很难稳定读取、写入和维护业务数据。

**核心判断**：销售系统、仓库管理、财务统计、项目交付、经营看板表面上是不同软件模块，背后其实都暗藏着同一类经营问题和机会：客户从哪里来、订单怎么流转、货和服务怎么交付、钱有没有收回来、利润到底在哪里、哪些动作可以被自动化。很多企业第一阶段并不需要重型 ERP 或复杂 SaaS。只要底层经营对象、数据流和联动关系设计得足够好，一个飞书多维表格就有机会承载一套统一的轻量经营系统。再通过飞书 CLI 和 AI Agent 连接，就可以继续做日报、经营分析、客户提醒、销售复盘、交付跟踪、爆款预测和自动化录入。

## 如何安装

#### 通用安装方式（适用于 Codex / Claude Code）

```bash
npx -y skills add guangxia321/cgx-baseos -g --all
```

#### Trae Solo / 手动安装

从 [GitHub Releases](https://github.com/guangxia321/cgx-baseos/releases) 下载最新的 `cgx-baseos-版本号.zip`。总包里包含 4 个独立的 skill zip：

1. `cgx-baseos.zip`
2. `cgx-baseos-diagnose.zip`
3. `cgx-baseos-build.zip`
4. `cgx-baseos-repair.zip`

如果工具要求一个 zip 对应一个 skill，就解压总包后逐个上传 `individual/` 里的四个 zip。

如果想本地构建，运行：

```bash
bash tools/build-skills.sh
```

构建产物在 `dist/skills/`。

## 如何更新

通过 `npx skills add` 安装的用户，重新运行一次同样的命令即可：

```bash
npx -y skills add guangxia321/cgx-baseos -g --all
```

手动安装的用户，从 [GitHub Releases](https://github.com/guangxia321/cgx-baseos/releases) 下载最新版，替换旧版本 skill。

## 工具箱

| Skill | 做什么 |
|---|---|
| `/cgx-baseos` | 主入口。根据用户问题判断进入诊断、构建还是维修 |
| `/cgx-baseos-diagnose` | 业务数据诊断。识别关键数据、重复动作、AI 可用数据和自动化机会，输出一页业务数据地图 |
| `/cgx-baseos-build` | 经营系统构建。基于业务数据地图，按四阶段搭建或重构飞书 Base |
| `/cgx-baseos-repair` | 维修排障。修复已有 Base、字段、公式、lookup、看板、筛选器和工作流问题 |

## 工具路径图

#### 从 0 开始做企业 AI 化

```text
diagnose（先判断哪些经营数据值得沉淀）
    ↓
business data map（一页业务数据地图）
    ↓
build（搭建最小经营系统）
    ↓
Feishu Base / low-code data system
    ↓
AI Agent（日报、经营分析、提醒、预测、自动化录入）
```

#### 已经有旧表或旧系统

```text
旧 Excel / 旧飞书 Base / 旧 SaaS 导出数据
    ↓
diagnose（判断哪些结构值得保留）
    ↓
build（重构为 AI 能读写的数据底座）
```

#### 已经搭好，但系统出错

```text
字段、公式、lookup、看板、筛选器、工作流异常
    ↓
repair（先定位最小对象）
    ↓
最小修复 + 样例数据验证
```

## 最小应用案例：经销商业务自动开单

笔者自己的经销商业务已经用这套思路做过一个最小系统：把客户、商品、订单、收款和财务流水先搭在飞书多维表格里，再通过 Codex 和飞书 CLI 连接这套数据。

现在开单不再只是手动填表，而是可以让 Agent 参与：

```text
用户输入开单需求
    ↓
Codex 理解客户、商品、数量、价格、收款方式
    ↓
飞书 CLI 写入飞书 Base
    ↓
自动生成订单记录
    ↓
同步补全或更新客户信息
    ↓
自动登记收款、应收或相关财务流水
```

这个案例的重点不是“自动开单”本身，而是证明一件事：只要底层数据结构设计得足够清楚，销售、客户、订单和财务就可以在同一套轻量系统里联动。后续要做经营日报、客户复购提醒、未收款追踪、商品销售分析或利润复盘，也都有了可被 AI Agent 调用的数据基础。

## 适合谁使用

### 1. 做企业服务、数字化服务或 AI 解决方案的人

如果你想帮客户做“企业 AI 化”，但客户没有稳定、开放、可被 Agent 调用的数据底座，可以先用广厦 BaseOS 快速搭出一套最小经营系统。

它适合用在客户交付的第一阶段：

- 先梳理客户的关键经营数据和高频流程。
- 用飞书多维表格搭一套可运行、可展示、可迭代的经营系统。
- 再把 AI Agent 接到这套数据底座上，继续做日报、分析、提醒、预测和自动化录入。

这比直接卖一个“大而全的 AI 系统”更容易落地：先让客户的数据结构跑起来，再逐步扩展 AI 能力。

### 2. 想自己搭经营系统的小企业老板

如果你是小企业老板，业务已经开始依赖客户、订单、项目、收款、费用、交付、库存或渠道数据，但这些数据还散落在微信、Excel、员工个人表格和旧系统里，也可以用广厦 BaseOS 先搭一个最小系统。

它适合用来验证：

- 哪些数据真的应该被沉淀。
- 哪些流程可以减少人工整理。
- 哪些指标应该先做成经营看板。
- AI 是否能基于这些数据帮你做查询、提醒、复盘和分析。

## 这个项目的边界

广厦 BaseOS 的定位不是做一个传统重型套件，而是用低代码数据工具搭建一套轻量、开放、可迭代的经营数据底座。

如果系统设计足够好，它可以在很多早期和中小企业场景中，替代一部分垂直 SaaS 或重型系统模块，例如轻量 CRM、销售管理、订单管理、项目交付、经营看板和基础进销存协同。

更准确地说，它不是按软件品类替代“某一个系统”，而是把企业第一阶段真正需要的经营对象和数据流重新组织起来：客户、商品或服务、线索、订单、项目、收款、费用、库存、交付、复购和经营指标。销售、仓库、财务、交付这些模块不应该一开始就被拆成互不相干的系统；它们应该先被看成同一条经营链路上的不同视角。只要这些对象和流程足够清晰，就可以先用飞书多维表格完成一套可运行的统一轻量经营系统。

但它不替代专业会计、报税、复杂供应链、复杂权限审批、完整制造业 ERP 或高度行业化的核心交易系统。这类场景可以后续对接或迁移，不应该在第一版最小系统里一次性做完。

它优先解决第一阶段最关键的问题：

- 企业有哪些关键经营数据。
- 哪些数据应该被结构化沉淀。
- 哪些流程最重复、最适合自动化。
- 哪些指标应该先进入看板。
- 人和 AI Agent 应该共用哪些录入、查询和分析入口。

第一版系统应该尽量小，但要足够稳定。先让数据跑起来，再逐步扩展。

## v1.0.0 包含什么

- 主入口 + 诊断 + 构建 + 维修的公开技能结构。
- 业务数据诊断和一页业务数据地图格式。
- 基础数据层、录入联动层、展示看板层、验收发布层四阶段构建流程。
- 销售系统、一人公司、企业服务、经营财务四类模板骨架。
- 看板、字段、公式、lookup、筛选器和工作流维修规则。
- 路由、诊断产物和四阶段构建的第一版测试用例。
- 构建脚本，可生成四个公开 skill zip 和统一整包。
